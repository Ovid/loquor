# Accessibility Audit: Loquor

**Date:** 2026-06-16 19:46:36
**Commit:** c30f882aa9e109f333c4caaa92fb132179f0abb4
**Platform(s):** Web (SPA)
**Tech stack:** React 19, Vite, custom CSS (no component library), WebLLM, ifvms.js Z-machine
**Files audited:** 15 production files (10 components + 5 CSS) plus `index.html`, `notices.ts`, `config.ts`, `useGameEngine.ts`, `loadError.ts`
**Existing a11y tooling:** none found (no `eslint-plugin-jsx-a11y`, no axe/pa11y, no a11y tests despite strict TDD elsewhere)
**Conformance target:** WCAG 2.2 AA (AAA noted as recommendations)
**Platform guidelines referenced:** N/A (web-only; WCAG applies directly)

## Executive Summary

Loquor is a screen-reader-inaccessible app today: the streaming game transcript — the entire content of an interactive-fiction game — is never announced (no live region anywhere in `src/`), and the sole command input has no accessible name. A blind player can launch the game but cannot read any response or know what the input does. Beyond those two complete barriers, the work is **partially Level A**: focus management on the download modal is absent, focus indicators are missing on core controls, localized FR/DE/ES content isn't marked with `lang`, and the light theme has several real contrast failures. The dark theme is in good shape. Estimated conformance: **fails Level A** (blocked on A1/A2/A5/A6), with a clear, mostly-mechanical path to AA.

## Impact Summary by User Group

- **Screen reader users:** Effectively cannot play. Game output is silent (no `role="log"`/`aria-live`), the command input is unlabeled, status/progress/NL-notice messages are never announced, and there are no landmarks or headings in the game view.
- **Low-vision users:** Light theme has multiple sub-AA contrast pairings (placeholder, location, controls, error text, borders); px font sizes ignore the OS font-size preference; no `prefers-contrast` path.
- **Colorblind users:** Mostly OK — the player echo carries a "you" text label and the command echo carries a `>` caret, so meaning isn't color-only in the critical spots. The pending-translation state leans on an opacity shimmer (weak), folded into the motion finding.
- **Motor-impaired users (keyboard/switch/sip-and-puff):** Download modal doesn't move/trap/restore focus and has no Escape, so focus escapes into the obscured game; missing focus indicators on the command input and modal buttons; modal action buttons are below the 24×24 target minimum.
- **Cognitive and learning disabilities:** Placeholder-only input label disappears on typing; download-failure notices give no recovery step and use unexplained "grammar-only" jargon; no persistent in-game help; localized content unmarked.
- **Deaf and hard-of-hearing users:** No barriers — the app has no audio/video content.
- **Vestibular and photosensitive users:** No `prefers-reduced-motion` support; two indefinite animations (caret blink, translation shimmer). No confirmed seizure risk (all flash rates measured under 3 Hz; the title flicker runs once and is opacity-only).

## Critical Issues (Complete Barriers)

### [C1] Game transcript is never announced to screen readers
- **File:** `src/ui/Scrollback.tsx:29-38` (the `<div className="scroll">`)
- **Barrier:** All streaming game output (room descriptions, responses, death messages) is appended to a plain `<div>` with no `role="log"` / `aria-live`. Verified: zero live regions exist anywhere in `src/`. The output IS the game; a screen-reader user hears nothing after typing a command.
- **Criterion:** 4.1.3 Status Messages — Level AA
- **Affects:** Blind/low-vision SR users — game is unplayable.
- **Fix:** Make the always-mounted scroll container a polite log:
  ```tsx
  <div className="scroll" ref={ref} role="log" aria-live="polite"
       aria-relevant="additions" aria-label="Game transcript" …>
  ```
  `role="log"` implies `aria-atomic="false"`; with each line its own `<p>`, only new paragraphs are announced.
- **Confidence:** High
- **Found by:** Screen Reader, React Platform (2 specialists)

### [C2] Command input has no accessible name
- **File:** `src/ui/CommandInput.tsx:48-62`
- **Barrier:** The `<input className="cmd">` is labeled only by `placeholder="type a command…"`. A placeholder is not an accessible name and vanishes on input. The only input mechanism of the game announces as an anonymous "edit text".
- **Criterion:** 4.1.2 Name, Role, Value — Level A; 3.3.2 Labels or Instructions — Level A
- **Affects:** All screen-reader users.
- **Fix:** Add `aria-label="Game command"` (and consider `enterKeyHint="send"`). Make the placeholder mode-aware ("type in English…" when NL is on) as a cognitive bonus.
- **Confidence:** High
- **Found by:** Screen Reader, Cognitive, React Platform (3 specialists)

## Serious Issues (Major Difficulty)

### [S1] Transient NL status (thinking / notices / queued) not announced
- **File:** `src/ui/Terminal.tsx:160-167`
- **Barrier:** `…thinking`, `nl-notice` (translation failed/timed out, drain notices), and queued lines are conditionally-mounted plain `<p>` with no live region. Even adding `aria-live` to a node mounted *with* its text typically won't announce — the region must exist empty, then be populated.
- **Criterion:** 4.1.3 Status Messages — Level AA
- **Affects:** SR users get no feedback during the multi-second NL translation or on failure.
- **Fix:** Render one always-present polite region and populate it:
  ```tsx
  <div aria-live="polite" className="nl-status">
    {nl.pending && <p className="nl-thinking">…thinking</p>}
    {nl.notice && <p className="nl-notice">{nl.notice}</p>}
  </div>
  ```
- **Confidence:** High
- **Found by:** Screen Reader, Cognitive, React Platform (3 specialists)

### [S2] Download modal has no focus management
- **File:** `src/ui/ModelDownloadModal.tsx:25-31`
- **Barrier:** `role="dialog" aria-modal="true"` but no focus move on open, no focus trap, no focus restore on close, no Escape handler. Focus stays behind the backdrop, so a keyboard user can Tab into the obscured game; `aria-modal="true"` misrepresents the actual containment.
- **Criterion:** 2.4.3 Focus Order — Level A (ARIA APG Dialog pattern)
- **Affects:** Keyboard, switch, sip-and-puff, and SR users.
- **Fix:** On open, store `document.activeElement`, focus the first action button (ref + `useEffect`); trap Tab/Shift+Tab within the dialog; restore focus on close; add `keydown` Escape → `onDecline`/`onCancel`. The logic already exists in `Landing.tsx:28-36` and can be mirrored.
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard/Motor, React Platform (3 specialists)

### [S3] Missing focus indicators on core controls
- **File:** `src/ui/components.css:176-185` (`.cmd { outline: 0 }`); `.themebtn` (18-34); `.enter` (`landing.css:229-246`); `.modal .sw` (`components.css:339-349`)
- **Barrier:** The command input removes its outline with no replacement (relies on caret-color only). The theme toggle, the "Light the lamp" CTA, and the modal action buttons have no `:focus-visible` style. (`.meta .sw`, `.vol`, `.dismiss` do have focus styles — so this is partial.)
- **Criterion:** 2.4.7 Focus Visible — Level A (2.4.11 Focus Appearance AA also relevant)
- **Affects:** Sighted keyboard / switch / voice-control users.
- **Fix:** Add a global fallback plus an override for the input:
  ```css
  :focus-visible { outline: 2px solid var(--brass); outline-offset: 2px; }
  .cmd:focus-visible { outline: 2px solid var(--brass); outline-offset: 2px; }
  ```
- **Confidence:** High
- **Found by:** Screen Reader, Keyboard/Motor (2 specialists)

### [S4] Localized FR/DE/ES content not marked with `lang`
- **File:** `index.html:2` (`<html lang="en">`); no per-element `lang` anywhere (transcript lines, `notices.ts`, picker labels)
- **Barrier:** When the player selects FR/DE/ES, translated output, notices, and option labels (Français/Deutsch/Español) render inside an `en` document. Text-to-speech voices the content with English phonetics.
- **Criterion:** 3.1.2 Language of Parts — Level AA
- **Affects:** SR users and read-aloud/dyslexia users in non-English mode. Directly undercuts the headline multilingual feature.
- **Fix:** Thread the active NL language into the display layer; set `lang={lang}` on localized transcript `<p>`, notices, and non-English picker labels. Per CLAUDE.md, apply across all four languages.
- **Confidence:** High
- **Found by:** Cognitive, Screen Reader (2 specialists)

### [S5] Light-theme placeholder text contrast fails badly
- **File:** `src/ui/components.css:186-189` (`.cmd::placeholder { color: var(--text-dim); opacity: 0.6 }`)
- **Barrier:** `--text-dim` #6b5740 @ 0.6 over #e7dcc4 ≈ **2.40:1** (light); dark ≈ **3.77:1** — both fail the 4.5:1 normal-text minimum. This placeholder is the input's only visible affordance.
- **Criterion:** 1.4.3 Contrast (Minimum) — Level AA
- **Affects:** Low-vision users.
- **Fix:** Drop `opacity: 0.6` and use a token that clears 4.5:1 (light `--text-dim` alone ≈ 5.04), or a dedicated `--placeholder` token meeting 4.5:1 in both themes.
- **Confidence:** High (hex/opacity verified; ratios trusted)
- **Found by:** Visual & Color

## Moderate Issues (Friction)

### [M1] No landmark regions
- **File:** `src/ui/App.tsx`, `Terminal.tsx`, `Landing.tsx` — entirely `<div>`-based
- **Barrier:** No `<main>`/`<header>`/`<nav>`; SR users can't jump to the transcript or game region.
- **Criterion:** 1.3.1 Info and Relationships — Level A
- **Fix:** Wrap the transcript in `<main>`, the status bar in `<header>`.
- **Found by:** Screen Reader

### [M2] Download `<progress>` unnamed; percentage/ETA not announced
- **File:** `src/ui/ModelDownloadModal.tsx:41-51`
- **Barrier:** `<progress value={pct} max={100}>` has no `aria-label`; the `{pct}% · ETA` text updates in a plain `<p>`. (The dialog itself is named via `aria-labelledby` — fine.) During a long download, SR users get no progress or completion cue.
- **Criterion:** 4.1.2 Name, Role, Value — Level A; 4.1.3 Status Messages — Level AA. *(Specialist's 1.1.1 citation dropped — `<progress>` is not non-text content.)*
- **Fix:** `<progress aria-label="Model download progress" …>` and wrap the percentage `<p>` in `aria-live="polite"`.
- **Found by:** Screen Reader, React Platform (2 specialists)

### [M3] Landing story-picker overlay has no Tab trap
- **File:** `src/ui/Landing.tsx:28-56`
- **Barrier:** The overlay focuses the dismiss button and handles Escape (good) but doesn't trap Tab — focus can move into the dimmed game behind it, and `CommandInput` auto-refocuses itself (`CommandInput.tsx:32-34`), actively pulling focus behind the overlay.
- **Criterion:** 2.4.3 Focus Order — Level A
- **Fix:** Trap Tab/Shift+Tab within `.plate`; suppress the CommandInput refocus effect while `picking` is true.
- **Found by:** Keyboard/Motor, React Platform (2 specialists)

### [M4] `.loaderr` error text contrast fails (hardcoded color)
- **File:** `src/ui/landing.css:261` (`color: #c2562f`)
- **Barrier:** Hardcoded (theme-blind) error color on a `role="alert"`: light ≈ 3.31:1, dark ≈ 4.36:1 — both under 4.5:1.
- **Criterion:** 1.4.3 Contrast (Minimum) — Level AA
- **Fix:** Replace with `var(--ember)` (light #a5371f ≈ 4.87:1); verify/bump dark `--ember` to clear 4.5:1.
- **Found by:** Visual & Color

### [M5] Light-theme control/location/echo/CTA contrast pairings under AA
- **Files & measured ratios (hex verified, ratios trusted):**
  - `.loc` brass #9a6312 on status gradient #e3d6ba→#d6c5a2 = **2.97–3.50:1** (`components.css:50-55`)
  - `.sw` brass-soft #7c5a2a on #d6c5a2 = **3.69:1** (`theme.css:34`, 13px)
  - `.nl-source` `--text` #3a2c1c @ 0.65 over #e7dcc4 = **3.87:1** (`components.css:192-195`)
  - `.enter` #f7efdd on brass #9a6312 = **4.40:1** (`landing.css:229-235`)
- **Barrier:** Normal-weight text below 4.5:1 in light theme. Dark theme passes.
- **Criterion:** 1.4.3 Contrast (Minimum) — Level AA
- **Fix:** Darken light `--brass`/`--brass-soft`; remove the `.nl-source` opacity dim (raise to ~0.78 or distinguish via the "you" label only).
- **Found by:** Visual & Color

### [M6] Border token `--rule` fails non-text contrast (both themes)
- **File:** `src/ui/theme.css:13` (dark #3a2c1c ≈ 1.46:1), `:42` (light rgba(90,65,35,0.4) ≈ 1.92:1)
- **Barrier:** Control borders on `.themebtn`, unselected `.vol`, `.chip`, `.nl-menu`/`.modal` rely on `--rule`, which is below the 3:1 boundary minimum — low-vision users can't perceive control edges.
- **Criterion:** 1.4.11 Non-text Contrast — Level AA
- **Fix:** Introduce a `--control-border` token ≥3:1 (dark ≈ #6a523a); keep decorative rules subtle.
- **Found by:** Visual & Color

### [M7] px font sizes ignore user font-size preference
- **File:** `src/ui/components.css`, `landing.css` (44 + 56 px declarations vs 8 rem, all in the modal)
- **Barrier:** Page zoom works, but px sizes don't honor the OS/browser default font size (a common low-vision accommodation). `.resume` 11px, `.howto`/`.statusbar` 13px are already small.
- **Criterion:** 1.4.4 Resize Text — Level AA
- **Fix:** Mechanical sweep px → rem (e.g. 15px → 0.9375rem), keeping visual size.
- **Found by:** Visual & Color

### [M8] No `prefers-reduced-motion` support
- **File:** all CSS (zero occurrences). Animations: `rise` per-paragraph (`components.css:143`), `blink` caret infinite (171), `xl-shimmer` infinite (221), `flicker` once (`landing.css:112`).
- **Barrier:** OS "reduce motion" is ignored. The per-line `rise` fires on every turn; the caret and shimmer pulse indefinitely (also implicates 2.2.2 Level A).
- **Criterion:** 2.3.3 Animation from Interactions — Level AAA; 2.2.2 Pause, Stop, Hide — Level A (indefinite animations)
- **Affects:** Vestibular / migraine users.
- **Fix:** Add one global block — but restore `opacity: 1` on `.scroll p` (its text starts at `opacity: 0` and only the animation reveals it, so naive disabling hides text):
  ```css
  @media (prefers-reduced-motion: reduce) {
    *, *::before, *::after { animation-duration: .001ms !important;
      animation-iteration-count: 1 !important; transition-duration: .001ms !important; }
    .scroll p { opacity: 1; animation: none; }
    .blink, .xl-pending { animation: none; }
    .xl-pending { opacity: .6; }
  }
  ```
- **Confidence:** High (no seizure risk: all flash rates < 3 Hz; title flicker runs once, opacity-only)
- **Found by:** Multimedia & Temporal

### [M9] Modal action buttons below 24×24 target minimum
- **File:** `src/ui/components.css:339-345` (`.modal .sw { padding: 0 }`); also `.meta .sw` (76-83)
- **Barrier:** Modal Download/Decline/Cancel buttons are ~17px tall (line-box only) — hard targets for touch / head-pointer / eye-gaze users. (The inline status-bar "⌄ Change story" likely qualifies for the 2.5.8 inline-text exception.)
- **Criterion:** 2.5.8 Target Size (Minimum) — Level AA
- **Fix:** `.modal .sw { padding: 6px 10px; min-height: 24px; }` (prefer 44px for dialog buttons).
- **Found by:** Keyboard/Motor

### [M10] "Unavailable" reasons exposed only via `title`
- **File:** `src/ui/NlLanguagePicker.tsx:50-61`
- **Barrier:** When NL is unavailable, the *why* is only in a `title` attribute — unreliable for SR, invisible on touch/keyboard.
- **Criterion:** 1.3.1 / 4.1.2 — Level A. *(Specialist's 1.4.13 dropped — that governs existing tooltips, not a missing one.)*
- **Fix:** Render the reason as visible localized text (or via `aria-describedby`).
- **Found by:** Screen Reader, Cognitive (2 specialists)

### [M11] ~~Auto-scroll overrides a player who scrolled up~~ — RETRACTED
- **File:** `src/ui/Scrollback.tsx:15-19`
- **Status: Retracted (not a real issue).** The command prompt is rendered inline at the bottom of the scroll container (`Terminal.tsx:168`, as `Scrollback` children), not a fixed bar. The natural workflow is scroll-up-to-read → scroll-back-down-to-type, so by the time you submit you're already at the bottom and the auto-scroll is a no-op. Zork is turn-based, so no output ever arrives unprompted to yank you mid-read. The original finding overstated a layout it hadn't verified.

### [M12] Download failure/stall notices not actionable + jargon — FIXED 2026-06-16
- **File:** `src/llm/notices.ts:112-136`
- **Barrier:** "Model download failed — staying grammar-only." gave no recovery step, and "grammar-only" was not just jargon but **inaccurate** — after a failure the NL phase reverts to `'off'` and typed input raw-sends to the Z-parser (`Terminal.tsx:173-174`); no deterministic grammar pipeline runs. The copy promised non-English players a native-language fallback that doesn't exist.
- **Criterion:** 3.3.5 Help — Level AAA.
- **Fix applied:** Rewrote both notices (EN/FR/DE/ES) to name the recovery path ("Pick a language again to retry, or keep playing by typing commands directly"), dropped the "grammar-only" jargon, and corrected the doc comments. Updated the pinned test assertions in `useModelDownload.test.tsx`; suite passes (19/19).
- **Open design question for Ovid (see note below):** whether a model-download failure *should* drop a non-English player all the way to raw English commands, or fall back to a real deterministic-NL mode.
- **Found by:** Cognitive
