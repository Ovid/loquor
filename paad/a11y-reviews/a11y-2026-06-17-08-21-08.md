# Accessibility Audit: Loquor

**Date:** 2026-06-17 08:21:08
**Commit:** 7e3a32ba02432af1f1e1a37289a99cfda74cadd0
**Platform(s):** Web
**Tech stack:** React 19, Vite 8, TypeScript, plain CSS (no component library), WebLLM, Vitest + Testing Library
**Files audited:** 17 (9 UI components, 5 CSS files, `index.html`, `notices.ts`, `loadError.ts` + supporting hooks/types)
**Existing a11y tooling:** None (no `eslint-plugin-jsx-a11y`, no `axe-core`; ESLint only runs `react-hooks` + `react-refresh`)
**Conformance target:** WCAG 2.2 AA (AAA noted as recommendations)
**Platform guidelines referenced:** ARIA Authoring Practices (APG) for the combobox and dialog patterns

## Executive Summary

Loquor's accessibility posture is **good and clearly already past one a11y pass** — the game transcript is a labelled `role="log"` live region, the modal has a correct focus trap (Tab containment + Esc + focus restore), icon-only controls hide their glyphs and carry text, most contrast pairs clear AA in both themes, and `prefers-reduced-motion` coverage is complete. No complete barriers (Critical) were found. The gaps that remain are **Serious** rather than blocking: NL status messages ("basic mode"/abstain/timeout) ride inside the transcript `log` region instead of a dedicated `role="status"`, the status-bar score/moves update silently for screen readers, and the natural-language input — the headline feature — is never signalled to the player. Estimated conformance: **partial AA** (a handful of AA gaps in status messaging, contrast, target size, and labels; no A-level structural failures beyond minor landmark/grouping issues).

## Impact Summary by User Group

- **Screen reader users:** Core gameplay works (transcript is a proper live region, picker/modal follow APG), but NL system feedback and score/moves changes can go unannounced, and a couple of controls have weak names. Serious friction, not a block.
- **Low-vision users:** Two light-theme contrast failures (the "…thinking" line and status-bar score/moves text); everything else clears AA. Zoom/reflow are safe (all text in rem/em).
- **Colorblind users:** The selected story on the landing screen is conveyed by border tint + glow with no shape/text cue. One real Use-of-Color issue.
- **Motor-impaired users (keyboard/switch/sip-and-puff):** Keyboard operability and the focus trap are solid. Main gap is sub-24px status-bar button targets.
- **Cognitive and learning disabilities:** Several notices use internal jargon or point at affordances the UI doesn't visibly show; the NL feature is undiscoverable; failed input is discarded and must be retyped.
- **Deaf and hard-of-hearing users:** N/A — the app is text-only with no audio/video.
- **Vestibular and photosensitive users:** Fully covered — every animation/transition respects `prefers-reduced-motion`; no flashing content; no player time limits.

## Critical Issues (Complete Barriers)

None found.

## Serious Issues (Major Difficulty)

### [S1] NL status messages are not in a region screen readers reliably announce
- **File:** `src/ui/Terminal.tsx:184-193` (thinking + notice), rendered inside `src/ui/Scrollback.tsx:30-78` (`role="log"`)
- **Barrier:** The "basic mode" / abstain / timeout notices and the "…thinking" indicator are children of the single `role="log" aria-live="polite" aria-relevant="additions"` transcript container — which also contains the interactive `CommandInput`. `role="log"` is tuned for sequential transcript prose, not status messages; there is no `role="status"` anywhere in the codebase. CLAUDE.md explicitly requires these notices to reach AT via an *appropriate* live region.
- **Criterion:** 4.1.3 Status Messages — Level AA
- **Affects:** Screen-reader users — they may type a sentence, get silently abstained (especially in FR/DE/ES), and hear nothing that the command failed or that they are in basic mode.
- **Fix:** Move `nl-notice` (and ideally the thinking indicator) into a sibling `<div role="status" aria-live="polite">` *outside* the `role="log"` container, and keep `CommandInput` out of the log region too. Add a test asserting the abstain text lands in `getByRole('status')`.
- **Confidence:** High
- **Found by:** Screen Reader, React Platform, Multimedia (three-way agreement)

### [S2] Score, moves, and location update silently each turn
- **File:** `src/ui/StatusBar.tsx:17-19` (status text from `src/ui/Terminal.tsx:153`)
- **Barrier:** The room location and score/moves (Z-machine v3 status line) render in plain `<span>`s with no `aria-live`. They change every turn; score/move count never appears in the transcript, so a blind player has no way to track them.
- **Criterion:** 4.1.3 Status Messages — Level AA (also 1.3.1)
- **Affects:** Screen-reader users — cannot track score/moves; may miss silent location changes.
- **Fix:** Wrap the score/moves value (and optionally location) in an `aria-live="polite"` region — wrap only the dynamic text spans, not the buttons.
- **Confidence:** Medium-High
- **Found by:** Screen Reader, React Platform

### [S3] Natural-language input is never signalled — the headline feature is undiscoverable
- **File:** `src/ui/CommandInput.tsx:54-55` (`aria-label="Game command"`, `placeholder="type a command…"`)
- **Barrier:** When an NL language is active the field accepts full English/FR/DE/ES sentences, but the label and placeholder say only "command", signalling rigid classic-parser syntax. The Landing how-to (`Landing.tsx:62-69`) describes classic syntax and is gone once in-game; it never mentions natural language. A player who could type "pick up the brass lantern" has no cue they may.
- **Criterion:** 3.3.2 Labels or Instructions — Level A
- **Affects:** All users (cognitive especially) — the core feature goes unused.
- **Fix:** When NL is on, reflect it in the placeholder/label or a persistent adjacent hint (e.g. "type plain English, e.g. open the mailbox"; localized for FR/DE/ES).
- **Confidence:** Medium-High
- **Found by:** Cognitive (two findings agreed)

## Moderate Issues (Friction)

### [M1] "improve" upgrade button has no contextual accessible name
- **File:** `src/ui/NlLanguagePicker.tsx:164-172`
- **Barrier:** No `aria-label`; accessible name is the bare "improve" (or "try the model anyway"). CLAUDE.md explicitly requires an `aria-label` on this control. "improve" alone gives no object/context.
- **Criterion:** 2.4.6 Headings and Labels — Level AA (also 4.1.2 A)
- **Affects:** Screen-reader users.
- **Fix:** `aria-label="Improve natural-language input (download AI model)"` (and a parallel label for the non-upgrade variant). Add a `getByRole('button', { name: ... })` test.
- **Confidence:** High
- **Found by:** Screen Reader

### [M2] Volume picker is exclusive selection built as toggle buttons, no group label
- **File:** `src/ui/Landing.tsx:76-89`
- **Barrier:** Three mutually-exclusive story choices use `<button aria-pressed>` toggles instead of a `radiogroup`; the visible "— choose your descent —" label is not programmatically associated as the group name.
- **Criterion:** 1.3.1 Info and Relationships — Level A (also 4.1.2 A)
- **Affects:** Screen-reader users — mutual exclusivity and arrow-key semantics not conveyed.
- **Fix:** `role="radiogroup"` + `aria-label`/`aria-labelledby` on the container, `role="radio" aria-checked` + roving tabindex on each; or keep buttons but add the group label and announce selection.
- **Confidence:** Medium-High
- **Found by:** Screen Reader

### [M3] Status-bar switch buttons below the 24×24 px target minimum
- **File:** `src/ui/components.css:90-97` (`.meta .sw { padding: 0 }`)
- **Barrier:** `.meta .sw` has `padding:0` at `0.8125rem`, so "Change story", the language trigger, and "improve" are text-height only (~13–17px). The modal already fixes this for its own buttons (`min-height:24px`) but the status-bar switches were left on the inline-text exception, which is shaky for standalone toolbar controls.
- **Criterion:** 2.5.8 Target Size (Minimum) — Level AA
- **Affects:** Motor-impaired, touch, eye-gaze, head-pointer users.
- **Fix:** Give `.meta .sw` (and `.nl-select-btn`) `padding: 4px 6px; min-height: 24px; display: inline-flex; align-items: center;`.
- **Confidence:** Medium-High
- **Found by:** Keyboard & Motor

### [M4] `.nl-thinking` "…thinking" text fails contrast (3.45:1, light theme)
- **File:** `src/ui/components.css:255-258`
- **Barrier:** `opacity:0.6` blends `--text #3a2c1c` over `--bg-base #e7dcc4` → effective `#7f725f` = **3.45:1**, below 4.5:1 in light theme. This is the same opacity-dim anti-pattern already fixed for `.cmd::placeholder` and `.nl-source`; `.nl-thinking` was missed. (Dark theme passes; `.nl-notice` at opacity 0.8 = 5.82:1 passes.)
- **Criterion:** 1.4.3 Contrast (Minimum) — Level AA
- **Affects:** Low-vision users in light theme during NL translation.
- **Fix:** Drop the opacity dim; use `color: var(--text-dim)` directly (clears 4.5:1 in both themes), matching the existing placeholder/nl-source fix.
- **Confidence:** High
- **Found by:** Visual & Color

### [M5] `.meta` score/moves text fails contrast over the lower status gradient (4.04:1, light)
- **File:** `src/ui/components.css:77-83`; gradient `src/ui/theme.css:49` (`#e3d6ba → #d6c5a2`)
- **Barrier:** `--text-dim #6b5740` over the darker gradient stop `#d6c5a2` = **4.04:1** (fail); over the lighter stop = 4.77:1 (pass). Text rendered across the bottom of the gradient fails. Dark theme passes (8.41:1).
- **Criterion:** 1.4.3 Contrast (Minimum) — Level AA
- **Affects:** Low-vision users in light theme.
- **Fix:** Darken `--text-dim` slightly in the light theme (e.g. `#63503a`), or flatten the status background so the worst-case stop clears 4.5:1.
- **Confidence:** Medium-High
- **Found by:** Visual & Color

### [M6] Selected story indicated by border + glow only (no non-color cue)
- **File:** `src/ui/landing.css:223-226` (`.vol.sel`)
- **Barrier:** The selected volume is distinguished only by `border-color: var(--brass)` + a `box-shadow` glow — no checkmark, label, or shape change. (The picker's listbox options at `components.css:317` *do* get a `✓`.)
- **Criterion:** 1.4.1 Use of Color — Level A
- **Affects:** Colorblind / low-vision users on the landing screen.
- **Fix:** Add a non-color cue to `.vol.sel` (checkmark/`▸`, "selected" text, or a thicker ring that reads as state). Pairs naturally with M2's radiogroup `aria-checked`.
- **Confidence:** Medium-High
- **Found by:** Visual & Color

### [M7] Untranslated English chips in a localized UI
- **File:** `src/ui/NlLanguagePicker.tsx:55` ("downloading… %"), `:61-63` ("· installed / · not installed"); `src/ui/Terminal.tsx:179-181` ("queued"), `:163` ("· basic")
- **Barrier:** These strings are literal English while the rest of the NL UI is localized via `notices.ts`. A FR/DE/ES player sees English chips mid-flow. (The "queued"/"you" chips are `lang="en"`-tagged so pronunciation is correct — the defect is untranslated *content*.) CLAUDE.md's "a fix in one language is a fix in all" applies.
- **Criterion:** 3.1.2 Language of Parts — Level AA (pronunciation satisfied; the gap is product/CLAUDE.md parity)
- **Affects:** Non-English players.
- **Fix:** Localize these chips through the same per-language mechanism as `notices.ts`.
- **Confidence:** Medium-High
- **Found by:** Multimedia, Screen Reader

### [M8] Failed input is discarded — full re-entry required after abstain/failure
- **File:** `src/ui/CommandInput.tsx:43-44` (clears on submit); failure notice `src/llm/notices.ts:55`
- **Barrier:** `setValue('')` runs on every submit; on an abstain/timeout the player's whole sentence is gone and must be retyped, even though the notice tells them to "try simpler wording".
- **Criterion:** 3.3.7 Redundant Entry — Level A (WCAG 2.2)
- **Affects:** Cognitive, motor, all users; worse for long sentences.
- **Fix:** On abstain/failure, restore the failed text into the field (or offer an "edit last" affordance) instead of discarding it.
- **Confidence:** Medium
- **Found by:** Cognitive

### [M9] Modal/overlay does not make background inert / `aria-hidden`
- **File:** `src/ui/ModelDownloadModal.tsx:99-106`, `src/ui/Landing.tsx:39-47` (trap: `src/ui/useFocusTrap.ts`)
- **Barrier:** The focus trap correctly contains **Tab** focus and handles Esc + restore, but neither modal renders in a portal nor marks the rest of the app `inert`/`aria-hidden`. `aria-modal="true"` is set (which should scope compliant SRs to the dialog), but support is uneven, so a screen-reader **virtual cursor** can still read the background.
- **Criterion:** 1.3.1 / 4.1.2 — Level A (APG Dialog)
- **Affects:** Screen-reader users using the virtual cursor.
- **Fix:** Apply `inert` (or `aria-hidden="true"`) to the app root while a modal/overlay is open; ideally render the dialog via a portal as the last child of `<body>`.
- **Confidence:** Medium
- **Found by:** React Platform

## Minor Issues & AAA Recommendations

- **[m1] Landing screen lacks a `<main>` landmark** — `src/ui/Landing.tsx:39-47`. 1.3.1 (A). Wrap the initial (non-overlay) landing in `<main>`. Found by Screen Reader.
- **[m2] Picker download chip is silent for screen readers** — `src/ui/NlLanguagePicker.tsx:53-55`. 4.1.3 (AA), low impact since the modal progress is correctly labelled. Add `aria-live="polite"` or rely solely on the modal. Found by Multimedia.
- **[m3] "·" middot prefix announced as literal noise; "basic" state not tied to the combobox** — `src/ui/NlLanguagePicker.tsx:160,163`. 1.3.1 (A). Wrap separators in `aria-hidden="true"`; associate state via `aria-describedby`. Found by Screen Reader.
- **[m4] "Couldn't translate" / first-miss notices use jargon and point at a non-visible affordance** — `src/llm/notices.ts:55-65,142-152`. 3.3.3 (AA). Reword to plain-language recovery ("try a shorter command like: go north"); surface quoting visibly if it's a real feature. Found by Cognitive.
- **[m5] Developer-facing load-error copy** ("Is the server running?", raw HTTP codes) — `src/ui/loadError.ts:15,19-23`. 3.3.3 (AA), delivered correctly via `role="alert"`; wording is the defect. Keep HTTP codes in the log. Found by Cognitive.
- **[m6] Arrow-key navigation doesn't `scrollIntoView` the active listbox option** — `src/ui/NlLanguagePicker.tsx:93-103`. 2.4.7 (AA), latent (5 options, no overflow today). Add `scrollIntoView({ block: 'nearest' })`. Found by Keyboard & Motor, React Platform.
- **[AAA] Consistent terminology** — "basic mode" / "the model" / "AI upgrade" / "improve" name the same concept differently (`notices.ts`, `NlLanguagePicker.tsx`, `ModelDownloadModal.tsx`). 3.2.4 (AA where it's the *same* control) / 3.1.3 (AAA). Pick one term and use it everywhere.
- **[AAA] Persistent in-game help** — help text exists only on the Landing and is gone once playing. 3.3.5 (AAA). A "?" control in the status bar would address it (and reinforce S3).
- **[AAA] `prefers-contrast` / `forced-colors` support** — none present (`src/ui/theme.css`). 1.4.6 (AAA) + Windows High Contrast robustness; in forced-colors the color-only selected-volume cue (M6) would also be lost.
