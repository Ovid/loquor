## Important Issues

### [I1] Superseded download clears the live download's stall watchdog (shared `stallTimerRef`)

- **File:** `src/llm/useModelDownload.ts:157,167` (timer ref declared `:62`; `clearStall` `:126-129`)
- **Bug:** `stallTimerRef` is a single ref shared across the hook's whole lifetime, and `clearStall()` runs **unconditionally** at the top of both the load's `.then` (`:157`) and `.catch` (`:167`), _before_ the `stale()` guard. Re-pick-language race: download #1 arms timer T1; the player re-picks â†’ download #2 aborts ac1 and `armStall()` stores T2 (the live timer) into the same ref; aborting ac1 makes load1 reject with `AbortError` (confirmed in `engine.fake.ts:42` and `engine.webllm.ts:65-67`), so load1's `.catch` fires a microtask later and calls `clearStall()` â€” which clears **T2**, the live download's watchdog.
- **Impact:** The re-picked download silently loses its no-progress protection â€” the exact "sits in `downloading` forever, manual cancel only" failure F6 was added to prevent. Triggers on a real user action (changing the language mid-download); recoverable only via manual cancel.
- **Suggested fix:** Make the stall timer per-load (capture the timeout id in a local each `requestDownload` and have each closure clear only its own id), or move `clearStall()` _below_ the `stale()` guard in both `.then`/`.catch` so a superseded load never touches the shared ref. The watchdog is one-shot and `cancelDownload` clears explicitly, so a genuine non-stale settle still clears correctly.
- **Confidence:** High
- **Found by:** Concurrency & State (verified)

### [I2] No unmount cleanup â€” stall timer fires on an unmounted tree and the in-flight load is never aborted

- **File:** `src/llm/useModelDownload.ts` (only `useEffect` is `:82-102`, whose cleanup `:99-101` covers just the `isCached` probe)
- **Bug:** Nothing clears `stallTimerRef` or calls `abortRef.current?.abort()` on unmount. If the hosting component unmounts mid-download (story swap / navigate away), the stall `setTimeout` survives and up to `DOWNLOAD_STALL_MS` later calls `setNotice`/`setInternal`/`ac.abort()` against an unmounted hook, and the in-flight `engine.load` (multi-GB fetch / VRAM) is never aborted.
- **Impact:** Post-unmount state updates (React dev warning â€” and under the project's "no stray console.warn" rule, a test-leak risk) plus a leaked fetch/timer. The `abortRef` half is pre-existing; F6 newly introduced the surviving _timer_ half, which is the worse one because it fires state setters.
- **Suggested fix:** Add `useEffect(() => () => { clearStall(); abortRef.current?.abort() }, [])` (hoist the clear logic so the unmount path can reach it).
- **Confidence:** Medium-High
- **Found by:** Logic & Correctness, Error Handling (verified)
