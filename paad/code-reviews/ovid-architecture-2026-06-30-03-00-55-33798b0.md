## Important Issues

### [C1] Production CSP `font-src 'self'` blocks the build's own inlined `data:` fonts
- **File:** `vite.config.ts:77` (the `font-src 'self'` entry in the `CSP` array)
- **Bug:** A production `vite build` inlines small self-hosted font subsets (under the 4096-byte `assetsInlineLimit`) as `url(data:font/woff2;base64,...)` in the bundled CSS — verified: `dist/assets/index-*.css` contains 6 such inline fonts across 4 `@font-face` blocks (JetBrains Mono **Cyrillic** `U+460-52F` and **Latin-Extended** subsets, woff2+woff, weights 400/500). The CSP allows `img-src 'self' data:` but `font-src 'self'` (no `data:`), so the browser blocks those `@font-face` loads in production.
- **Impact:** Those glyph subsets fail their self-hosted load and fall back to system fonts — degraded, inconsistent rendering — only in production, and **only this CSP causes it**. The status reason's claim that "the base (non-LLM) game cannot be broken by this policy" is therefore wrong. It evades the F-p manual checklist, which exercises only the LLM download path, never font rendering. (Note: the review's original `ka`/Georgian framing was corrected during verification — no Georgian `U+10A0-10FF` subset is bundled; Georgian already renders via system fallback. The real regression is Cyrillic + Latin-Ext.)
- **Suggested fix:** `"font-src 'self' data:"` (lowest-risk; `data:` fonts are same-document, not an exfiltration vector). Alternatively set `build.assetsInlineLimit: 0` so nothing is inlined and `'self'` suffices. Either way, add "render a Cyrillic transcript line, confirm no `font-src` CSP violation" to the F-p checklist.
- **Confidence:** High (90 — build-verified)
- **Found by:** Security

### [C2] `deleteModel` during an in-flight re-download can resurrect the just-deleted model
- **File:** `src/llm/useModelDownload.ts:398-413` (`deleteModel`) + `:256-277` (the load `.then`); `src/llm/engine.webllm.ts:102-105` (`unload` doesn't bump `loadEpoch`), `:138-142` (`deleteCache`)
- **Bug:** `deleteModel` calls `engine.deleteCache?.()` but never calls `abortInFlight()`, and neither `deleteCache()` nor `unload()` aborts `abortRef.current` or bumps `loadEpoch`. So an in-flight `engine.load().then()` can resolve *after* the delete; its `stale()` guard (`ac.signal.aborted || abortRef.current !== ac`) is false because the delete never touched `ac`, so it re-runs `setInstalled(true)` + `setInternal({phase:'on', model:'full'})`.
- **Reachability (verified):** During a **re-download/upgrade of an already-cached model** (`requestUpgrade` → `requestDownload`), `requestDownload` sets `phase:'downloading'` but leaves `installed` **true** (only the boot probe and a successful load set it true; only `deleteModel` sets it false). `PreferencesModal.tsx:206` gates the "Delete cached model" row on `modelInstalled = nl.installed`, and the download modal and Preferences modal are independent in `Terminal.tsx`, so the delete row is rendered and clickable while the load is in flight. The two then interleave.
- **Impact:** End state `installed=true`, `phase:on/full`, but the on-disk cache is wiped and the engine was unloaded mid-load — the model reports installed/full yet is unusable until a page reload (the boot probe self-corrects only then). Defeats the player's explicit disk-reclaim. No test covers this interleaving (`useModelDownload.test.tsx` tests delete in isolation only).
- **Suggested fix:** In `deleteModel`, call `abortInFlight()` *before* `engine.deleteCache?.()` (kills the stall timer + aborts `ac`, so the load's `.then()` goes stale). Belt-and-braces: bump `loadEpoch` in `unload()`/`deleteCache()` so a late `load()` that hasn't yet assigned `this.engine` releases instead of installing. Add `abortInFlight` to the `useCallback` deps, and add a test that injects a rejecting/late `deleteCache` and asserts no resurrection (and that it spy-asserts the `log.warn` path, per the no-leak convention).
- **Confidence:** High (80)
- **Found by:** Error Handling & Edge Cases + Concurrency & State (cross-specialist agreement)
