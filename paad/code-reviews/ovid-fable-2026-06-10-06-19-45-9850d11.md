## Important Issues

### [D] Lexicon verb targets in `verbSynonyms` (and truncated `inflat`) are dead at the deterministic stage; the roundtrip gate hides it

- **File:** `src/llm/lexicon/parse.ts:109-114` (`verbArityOk`), `src/llm/lexicon/roundtrip.test.ts:43-53`
- **Bug:** `verbArityOk` checks only `verbsOnly`/`movement`/`verbs1`/`verbs2`. The verbs `break`, `touch`, `taste`, `hide`, `shout`, `sit`, `remove` exist **only** in `verbSynonyms` in all three generated vocabs, and `inflate` exists in no list at all (the vocab stores the Z-machine 6-char-truncated `inflat`). Every fr/de/es lexicon entry targeting them is dead: runtime-verified `parseLexicon('casse la bouteille', ‚Ä¶)` ‚Üí miss, `'touche le sac'` ‚Üí miss, `'gonfle le bateau'` ‚Üí miss (the UAT F-R trap word itself). The roundtrip test passes because it deliberately unions `verbSynonyms` and applies truncation-widening into its gate set ‚Äî checking a property the runtime parser doesn't have.
- **Impact:** Defeats v2's deterministic-first design for a whole verb class (misses fall to the slow, fallible LLM), and the CI gate certifies dead data, so future additions rot silently.
- **Suggested fix:** Accept `vocab.verbSynonyms` in `verbArityOk` (synonyms inherit their head verb's arity in ZIL; a wrong-arity emit just earns the Z-parser's orphan prompt) and add truncation awareness, **or** retarget the lexicon data and tighten the roundtrip gate to mirror `verbArityOk` exactly.
- **Confidence:** High (runtime-proved)
- **Found by:** Logic-B; corroborated by Error-Handling-B; Verifier resolved the contradiction with Contract-B (whose diagnostic was fooled by the test's widened gate) in Logic-B's favor

### [H] Whole-remainder and pronoun paths gate on arity-1 only; verbs2-only verbs (attack, kill, give, throw, tie, ‚Ä¶) deterministically miss

- **File:** `src/llm/lexicon/parse.ts:200-205` (whole-remainder), `:179` (pronoun), vs `:221-225` (personal-a, correctly widened)
- **Bug:** `attack, kill, give, throw, tie, cut, burn, lock, unlock, put, turn` are verbs2-only in all three vocabs. The whole-remainder one-object path requires `verbArityOk(verb, vocab, 1)`, so `attaque le troll` / `toete den troll` always miss ‚Äî while the structurally identical Spanish personal-a path was explicitly widened to `arity 1 OR 2` with a comment naming exactly this verb class. Runtime-verified: `parseLexicon('attaque le troll')` ‚Üí miss; `ataca al trol` ‚Üí command.
- **Impact:** Large deterministic-coverage loss for combat/manipulation verbs in FR/DE (8‚Äì11s LLM round-trip with mistranslation risk, language-dependent behavior). The cross-language inconsistency demonstrates it's an oversight, not the documented strictness rule.
- **Suggested fix:** Apply the same `verbArityOk(‚Ä¶,1) || verbArityOk(‚Ä¶,2)` allowance to the whole-remainder and pronoun paths.
- **Confidence:** High
- **Found by:** Error-Handling-B

### [C] Stage-4 vocab passthrough validates a punctuation-stripped line but sends the unstripped original

- **File:** `src/llm/translate.ts:130-139` vs `src/llm/useNaturalLanguage.ts:412-417`; same shape at stage 3 meta (`:395-399`)
- **Bug:** `isVocabPassthrough` strips `[!.?,;:]+$` before tokenizing, but the hook sends `clause.trim()` ‚Äî punctuation intact. `splitClauses` consumes only `.`/`;`, so `!?,:` survive. Zork I's dictionary word separators are exactly `. , "` (verified from the story header), so `take lamp!` is classified "all parser-known words", sent verbatim, and the game replies "I don't know the word lamp!". Regression vs main: `north!` previously reached `parseDirection` (which normalizes) or the LLM; the new passthrough stage intercepts it first and stage 5 never runs.
- **Impact:** Enthusiastic punctuation burns a game turn on a parser error in exactly the stage built to prevent that (the F-H killer); the deterministic stage produces a worse result than falling through.
- **Suggested fix:** Send the same normalized text the check validated (have `isVocabPassthrough` return it, or strip once in `runClause` for both gate and send); do the same for the stage-3 meta send.
- **Confidence:** High
- **Found by:** Error-Handling-A, Error-Handling-B, Contract-A

### [E] Container-anaphora pronouns with ON semantics emit `in`

- **File:** `src/llm/lexicon/parse.ts:195`; data at `fr.core.ts:260` (`dessus`), `de.core.ts:293` (`darauf`, `drauf`), `es.core.ts:367` (`encima`)
- **Bug:** The container-anaphora branch hardcodes `in`, but `pronounsContainer` mixes "inside it" words with "on top of it" words. Runtime-verified: `pose le livre dessus` (antecedent `table`) ‚Üí `drop page in table`; `mets le livre dessus` ‚Üí `put page in table`. The player said ON; put-in vs put-on is gameplay-meaningful in Zork (surfaces refuse `in`).
- **Impact:** Confidently wrong command from the never-guess deterministic stage; the LLM never gets a chance. Tests cover only `dedans`, so it's silent.
- **Suggested fix:** Carry the preposition in the data (`{ word, prep: 'in' | 'on' }`) and emit it, or drop the ON-pronouns so those inputs miss to the LLM.
- **Confidence:** High
- **Found by:** Logic-B, Contract-B

### [F] German `an: 'on'` breaks tie/attach constructions ‚Äî the rope puzzle fails deterministically for German players

- **File:** `src/llm/lexicon/de.core.ts:259`, consumed by the prep split at `parse.ts:229-241`
- **Bug:** German `an` in `binde X an Y` means English "to". Runtime-verified: `binde das seil an das gelander` ‚Üí `tie rope on railing`, but Zork's only two-object tie syntax is `TIE OBJECT TO OBJECT` (`zork1/gsyntax.zil:497`). Because the lexicon stage succeeds, the LLM never sees it.
- **Impact:** The Dome Room rope puzzle (and give-to constructions) deterministically fail in German.
- **Suggested fix:** Map `an` ‚Üí `to` (the attach sense dominates two-object commands; `auf` already covers "on"), or make the prep split retry alternate senses / fall through to the LLM.
- **Confidence:** High
- **Found by:** Logic-B

### [G] `extractNouns` first-wins canonical dedupe silently drops later objects' dictionary words

- **Confidence:** High
- **Found by:** Logic-B, Error-Handling-B

### [M] Unbounded lazy `engine.load` inside the translate critical section can wedge all input permanently

- **File:** `src/llm/useNaturalLanguage.ts:334` (load awaited before the watchdog exists), `:362` vs `:373` (queue check precedes phase check), `src/ui/Terminal.tsx:188`
- **Bug:** On a reload session with the pref restored to 'on' (model cached on disk, not memory), the first command triggers `engine.load` inside `generateRaw` with no timeout ‚Äî the watchdog covers only `generate`. If the load stalls (WebGPU init, partial cache eviction ‚Üí network), `translatingRef` stays true forever: every line queues to the cap then drops with "Queue full". Because the queue check runs before the phase check, switching NL off doesn't restore raw play ‚Äî and `disabled={nl.pending && nl.state.phase !== 'on'}` then disables the input box entirely.
- **Impact:** A single stalled load deadlocks all player input for the session; only a page reload recovers.
- **Suggested fix:** Bound the lazy load (its own generous timeout or include it under a larger watchdog budget) and check `internal.phase` before the queue check so 'off' always restores raw play.
- **Confidence:** High
- **Found by:** Error-Handling-A, Concurrency-A

### [N] Mid-drain language switch is ignored ‚Äî queued lines translate under the old language; "off is instant" isn't for the queue

- **File:** `src/llm/useNaturalLanguage.ts:373, 381` (closure captures `internal`/`lex`/`activeLang`/`grammar` once), drain loop `:707-780`, comment at `:308`
- **Bug:** The drain loop runs inside one `translate` closure but consumes the shared `queueRef`. - **Confidence:** High

### [O] Story switch mid-drain fires old-game translations into the new game

- **File:** `src/ui/App.tsx:80-86` (Terminal not keyed by slug), `src/ui/Terminal.tsx:115-119` (`sendLine` reads `engineRef.current` at call time), `src/llm/useNaturalLanguage.ts:171-174` (vocab effect resets tracker only ‚Äî queue never cleared)
- **Bug:** "Change story" is reachable during play (StatusBar ‚Üí Landing overlay) and swaps `storyBytes` on the mounted Terminal. An in-flight drain keeps the old vocab/lexicon/tracker closure, but `sendLine`/`awaitTurn` resolve to the NEW engine ‚Äî old-game queued lines, translated with the old game's vocabulary, fire turns into the freshly booted game. - **Suggested fix:** `key={slug}` on Terminal to remount per story, or an epoch token bumped on signature/vocab change that the drain checks between lines (abandoning and clearing `queueRef` when it changes).
- **Confidence:** High
- **Found by:** Concurrency-A, Concurrency-B

### [K] `localStorage` default parameter defeats nlpref's try/catch ‚Äî successful download reported as failure in cookie-blocked browsers

- **File:** `src/llm/nlpref.ts:13, 37-38`; consequences at `src/llm/useNaturalLanguage.ts:271-287, 304-323`
- **Bug:** `store: Storage = localStorage` evaluates at call time, before the body's try is entered; in Chrome with "block all cookies" the `localStorage` getter itself throws SecurityError. `requestDownload`'s `.then` calls `writeNlPref` after `setInternal({phase:'on'})` ‚Äî the throw lands in the `.catch`, which shows "Model download failed ‚Äî staying grammar-only." and flips phase off **after the download succeeded**. `setLanguage`/`declineDownload` throw uncaught from UI handlers.
- **Impact:** Silent misclassification of success as failure plus uncaught exceptions on every language change, in a real browser configuration the inner try/catch was clearly written to handle.
- **Suggested fix:** Resolve the store inside the try (`store?: Storage` ‚Üí `store ?? localStorage` within the body), here and anywhere the pattern repeats (`useTheme.ts` has it with no try at all).
- **Confidence:** High
- **Found by:** Error-Handling-A

### [L2] Cancelled model download keeps running; concurrent `load` calls race on `this.engine`, leaking an engine or leaving phase 'on' with no engine

- **File:** `src/llm/engine.webllm.ts:30-55`, `src/llm/useNaturalLanguage.ts:247-249, 290-293`
- **Bug:** `cancelDownload` aborts the signal, but `WebLlmEngine.load` never plumbs it into `CreateMLCEngine` ‚Äî the old download/init continues. Re-picking a language starts a second concurrent `load` (no guard); `this.engine` is assigned **before** the abort check, so a late-resolving cancelled load clobbers the fresh engine then `unload()`s, leaving the fresh MLCEngine unreferenced (VRAM leak) and/or the hook in phase 'on' with `isLoaded() === false` (next translation triggers a third full load).
- **Impact:** Double VRAM allocation / GPU OOM on exactly the constrained devices the capability gate worries about; state machine and engine residency disagree.
- **Suggested fix:** Abort the previous controller in `requestDownload` before replacing it; add an in-flight-promise/epoch guard in `WebLlmEngine.load` and unload the loser of the race instead of leaking it.
- **Confidence:** Medium-High
- **Found by:** Concurrency-A (verifier confirmed the cancel‚Üíre-pick path; the picker-during-download path is not reachable)
