## Important Issues

### [I1] `reduce()` clears the input request on any non-line/char input event

- **File:** `src/glkote-react/reduce.ts:182-190`
- **Bug:** When a GlkOte `update` carries a non-empty `input` array that contains no `line` or `char` request (e.g. a `hyperlink` or `mouse` event request, both valid GlkOte shapes), `update.input.find(i => i.type === 'line' || i.type === 'char')` returns `undefined` and `inputRequest = req?.type ?? null` forces `inputRequest` to `null`. The code deliberately distinguishes `length === 0` ("request cleared") from non-empty, then collapses the non-empty-but-unmatched case back to `null` anyway.
- **Impact:** The input box would be disabled even though the VM is still genuinely waiting for input. **Latent today** — Zork I–III only ever issue line input, so this path is never hit by the shipped games. It becomes a real bug only if a non-line/char request is ever produced.
- **Suggested fix:** When `input` is non-empty but has no line/char request, leave `inputRequest` unchanged rather than nulling it: `const req = update.input.find(...); if (req) inputRequest = req.type as 'line' | 'char'` (drop the `?? null`).
- **Confidence:** Medium (80)
- **Found by:** Error Handling & Edge Cases (root cause shared with [S1], flagged by 3 specialists)

## Suggestions

- **[S1] `charIsMore` desyncs from `inputRequest` on content-only updates** — `src/glkote-react/bridge.ts:96` recomputes `charIsMore` every `update()`; a content-only update (no `input` field) sets `req` undefined → `charIsMore` resets to `false`, while the reducer leaves `inputRequest` at `'char'`. Real desync, latent because Zork never emits char requests. Gate the recompute on `arg.input !== undefined`. _(Error Handling + Logic-Core + Concurrency — 3 specialists; conf 72)_
- **[S2] `restore_allstate` is dead code with a misleading contract comment** — `src/glkote-react/bridge.ts:126-128` / `src/glkote-react/types.ts:54`. The vendored `glkapi.js` never calls `GlkOte.restore_allstate()`; it replays autorestore state via the `update()` `autorestore` field instead. Harmless, but the comment implies a symmetric save/restore round-trip that does not happen. Remove the method or correct the comment. _(Contract & Integration; conf 85)_
- **[S3] Untrimmed command sent to the VM** — `src/ui/CommandInput.tsx:37-39`. The empty-guard tests `value.trim()` but `onSubmit(value)` forwards the raw, untrimmed string. `onSubmit(value.trim())` matches what was validated. Low impact (the Z-machine tokenizer tolerates leading space). _(Logic-UI; conf 88)_
- **[S4] `classify()` mis-tags short responses as room headings** — `src/glkote-react/reduce.ts:117-120`. The regex `/^[A-Z][^.!?]{2,40}$/` matches `"Opened"`, `"Dropped"`, `"Taken"`, `"Done"` and styles them with the room CSS class. Purely cosmetic; real Zork acks usually carry a period, which the regex excludes. _(Error Handling; conf 82)_
- **[S5] Unvalidated theme read from `localStorage`** — `src/ui/useTheme.ts:7-15`. `localStorage.getItem(KEY) as Theme` trusts any non-null value; a corrupt value like `"Light"` is persisted and makes the first toggle click misbehave. Validate: `return v === 'light' ? 'light' : 'dark'`. _(Logic-UI; conf 80)_
- **[S6] `disabled` prop on `CommandInput` is dead; no dead-VM submit guard** — `src/ui/CommandInput.tsx:5,47` accepts and applies `disabled`, but `Terminal.tsx:58-68` never passes it (always `false`). Submitting after a true game-end calls `sendLine` against a dead VM (a no-op via stale `gen`, so low harm). Either wire `view.ended` or remove the prop. _(Logic-UI; conf 78)_
- **[S7] `flushAutosave` dead-spins 500 ms after a clean quit and isn't a true settle barrier** — `src/zmachine/engine.ts:153-160` polls `dialog.hasSave()` 100×5 ms; after clean-quit CLEAR `hasSave` is always `false`, so it runs the full 500 ms. `hasSave` does a fresh `idbGet` and bypasses `writeChain`, so it confirms presence, not drained writes. Prefer awaiting `dialog.flushWrites()`. Test/flush-only helper, minimal impact. _(Concurrency; conf 75)_
- **[S8] Per-instance `writeChain` doesn't order cross-engine writes to the same key** — `src/storage/dialog.ts:70` + `src/ui/Terminal.tsx:24-33`. During the StrictMode double-mount window, two `IdbDialog`s write `autosave:<sig>` via separate IndexedDB connections with no mutual ordering. Benign today (identical initial snapshot; throwaway disposed before it runs a turn); would reintroduce a stale-snapshot race if a throwaway ever ran a turn. Consider a module-global, key-scoped write chain. _(Concurrency + Logic-Core; conf 72)_
- **[S9] Signature derived from raw bytes diverges from ifvms for Blorb files** — `src/zmachine/signature.ts:4-13`. ifvms keys autosave from `origram` (extracted Z-code), while `signature()` hashes the first 30 raw bytes. Identical for raw `.z3` (the only format `src/games/catalog.ts` ships), so latent — but a future `.zblorb` would silently never resume. Assert raw Z-code, or derive the signature post-`identify()`. _(Logic-Core; conf 80)_
- **[S10] `ViewState.ended` latches permanently** — `src/glkote-react/reduce.ts:144-202`. `ended` is seeded from `prev.ended` and only ever set `true`; an in-place RESTART after death never resets it. No consumer currently gates on it (Terminal remounts with a fresh view), so it's a latent trap for any future UI that reads `ended`. Reset on a fresh line-input request or a `clear:true` screen wipe. _(Concurrency; conf 70)_
- **[S11] `dispose()` does not cancel already-enqueued in-flight IndexedDB writes** — `src/storage/dialog.ts:79-90` + `src/zmachine/engine.ts:168-172`. `disposed=true` no-ops _future_ writes but an in-flight `idbSet` still settles. Benign today (throwaway and live engine write the same key/initial state); the surrounding comments imply stronger guarantees than the code provides. _(Logic-Core; conf 70)_
- **[S12] No Content-Security-Policy** — `index.html:3-8`. Defense-in-depth only; there is no current injection vector. A `default-src 'self'` CSP would make the no-CDN / no-exfiltration privacy promise fail closed against future regressions. _(Security; conf 65)_
