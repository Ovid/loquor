## Important Issues

### [I1] Modified-quantifier deterministic path is English-only — FR/DE/ES parity gap

- **File:** `src/llm/lexicon/parse.ts` (`resolveEnglishQuantifierPhrase`) + wiring in `src/llm/translatePipeline.ts` (English `else` branch); fr/de/es path at `src/llm/lexicon/parse.ts:374`
- **Bug:** English now resolves _modified_ quantifiers deterministically — "put all in case", "drop all but the lamp", "take everything except the lamp" raw-send to the Z-parser. The fr/de/es `quantifiersAll` branch in `parseLexicon` fires only for the **bare single-token** remainder (`tokens.length === 1`), so "mets tout dans la caisse" / "leg alles in die Kiste" / "pon todo en la caja" skip it, fail noun resolution ("tout/alles/todo" is not a noun), and fall through to the LLM. There is no fr/de/es analogue of `resolveEnglishQuantifierPhrase`.
- **Impact:** The branch's own BUG F comment documents that the LLM mangles the equivalent English ("put all in case" → "take large bag"). FR/DE/ES players typing the natural localized endgame treasure-casing shortcut get exactly the failure the English fix was added to prevent. They have an LLM net (not a crash), but this is the precise asymmetry CLAUDE.md's "a fix in one language is a fix in all of them" rule targets.
- **Suggested fix:** Generalize the quantifier handling in `parseLexicon` to a multi-token form mirroring `resolveEnglishQuantifierPhrase` — when the post-verb remainder leads with a `quantifiersAll` word followed by parser-known tokens, emit `<verb> all <rest>`, mapping the localized prep via `core.preps` and the localized BUT/EXCEPT. If deferring fr/de/es to the LLM here is intentional, state the reason explicitly in code/commit (the steering rule requires the reason be written, not implicit) — and note this is a player-experience call that, per CLAUDE.md, is worth surfacing to Ovid.
- **Confidence:** High
- **Found by:** Multilingual parity specialist (verified)

### [I2] English `else` branch is gated on `lex?.nouns` rather than on the active language

- **File:** `src/llm/translatePipeline.ts` — `runClause`, `if (lex?.nouns) { … } else { <English resolvers> }`
- **Bug:** The `else` runs the English-mode resolvers (`resolveEnglishPronoun`, `resolveEnglishQuantifier`, `resolveEnglishQuantifierPhrase`, `isEnglishPronounClause`), but its condition means "this game has no noun lexicon," not "the active language is English." `lex` can be non-null with `lex.nouns === null` (`nounLexicon(lang, sig)` returns null for an unregistered signature; `useNaturalLanguage.ts` comments "nouns may be null for an unknown signature"). A fr/de/es player on a game whose noun lexicon is unregistered would have non-English input run through the English resolvers.
- **Impact:** Currently **latent/unreachable** — all three Zork signatures are registered for fr/de/es, and the English matchers require English-only tokens (`it`/`them`/`all`/`everything`), so a misfire is unlikely even if reached. But the contract is now "this `else` == English" while the gate actually says "no noun lexicon"; a future game or lexicon change silently turns that mismatch into wrong behavior.
- **Suggested fix:** Gate the new English block explicitly on the language, e.g. `} else if (activeLang === 'en') {` (`activeLang` is already a `runClause` parameter), leaving a non-English-with-null-nouns clause to fall through to the LLM as before.
- **Confidence:** Medium (real contract mismatch; impact latent)
- **Found by:** Logic & Correctness specialist (verified)
