## Important Issues

### [I1] resolve()'s cache read & engine-load check live inside the engine-failure catch — transient errors leak to console AND skip the LLM fallback
- **File:** `src/translate/useOutputTranslation.ts` (the `resolve` try/catch — `cacheGet` first await, `engine.isLoaded()` check before the gate, catch at end)
- **Bug:** The first `await cacheGet(...)` sits inside the `try` whose `catch` treats any non-`ExpectedXlateStop` throw as a broken engine (`console.error('[xlate] …')` + settle English). `cacheGet → idbGet → tx` **rejects** on transient IndexedDB errors (`onerror`/`onabort`/open-`blocked` in `src/storage/idb.ts`), which is distinct from a cache *miss* (`undefined`). So a cache-read error is (a) misreported as an engine failure on the console, and (b) **bypasses the LLM fallback entirely**. Separately, `engine.isLoaded()` is checked *before* `gate.run`, so an unload while the task is queued makes `engine.generate` throw `'engine not loaded'` → also a non-`ExpectedXlateStop` `console.error`. The backlog path correctly `.catch(() => {})`s the same `cacheGet`; this live path does not.
- **Impact:** Foreseeable paths (IDB pressure/private mode, engine teardown during a queued translation) both **violate the enforced "tests/output stays pristine" rule** — `src/test/setup.ts` throws on any stray `console.error` — and the cache-error case needlessly downgrades a translatable line to English.
- **Suggested fix:** Wrap the initial `cacheGet` in its own `try/catch` that treats a read failure as a miss (fall through to the fallback). Re-check `engine.isLoaded()` *inside* the gate body and throw `ExpectedXlateStop` (or reclassify the `'engine not loaded'` error) so engine-teardown degrades silently to English.
- **Confidence:** High
- **Found by:** Error Handling (E1), Logic (L1), Concurrency (C2-area) — three specialists converged on this `resolve()` root.

### [I2] EngineGate hands the gate to the next task before the aborted generation has actually settled — two generations can overlap on the single engine
- **File:** `src/translate/useOutputTranslation.ts` (the `gate.run('output')` body), `src/llm/useNaturalLanguage.ts` (`generateRaw` inside `gate.run('input')`), `src/llm/engine.webllm.ts` (`generate` / `onAbort → interruptGenerate`)
- **Bug:** When the watchdog wins `Promise.race`, the gate-task body rejects and `EngineGate.run`'s `finally` immediately calls `next.start()`. But the orphaned `engine.chat.completions.create()` is not awaited — `ac.abort() → interruptGenerate()` only *requests* the worker to stop. The next waiter can therefore call `create()` on the same shared engine before the previous call finishes interrupting, breaking the gate's mutual-exclusion invariant precisely on the timeout/preempt path the gate exists to handle. The gate's own tests only resolve tasks cleanly, so this passes CI while being unsound.
- **Impact:** Overlapping `create()` calls on a non-reentrant WebLLM engine can corrupt decode state, throw, or wedge. Bounded to the (rarer) watchdog/preempt path, hence Important rather than Critical.
- **Suggested fix:** The gate task must not release until the underlying `create()` has truly settled — e.g. await the orphaned generate's settlement (`await gen.catch(() => {})`) before returning from the gated fn, or have `WebLlmEngine.generate` resolve only once the engine is genuinely idle post-interrupt. Fix the input and output call sites together (shared shape).
- **Confidence:** Medium
- **Found by:** Concurrency (C1)

### [I3] useOutputTranslation fallback effect has no cleanup — in-flight resolve() runs setOverlay after unmount, and the queued gate task is never abandoned
- **File:** `src/translate/useOutputTranslation.ts` (the fallback effect returns no cleanup; `epochRef` bumps only on `[corpus, lang, signature]`, never on unmount)
- **Bug:** `void resolve(...)` launches a detached async chain ending in `settle → put → setOverlay`. On unmount nothing bumps `epochRef` or aborts the in-flight `AbortController`, so a queued/in-flight gate task still runs a full LLM generation and calls `setOverlay` on an unmounted component. `settle`'s epoch and basis guards don't trip on unmount.
- **Impact:** Wasted post-unmount state update (React routes the unmounted-update / `act(...)` warning through `console.error`, again colliding with the pristine-output guard) and wasted GPU on a queued generation nobody renders after the user leaves the game / on HMR.
- **Suggested fix:** Return a cleanup from the effect that bumps `epochRef.current++` (and ideally aborts the in-flight controller); the existing epoch guard in `settle` then makes late resolutions no-ops and queued tasks hit the abandon check.
- **Confidence:** Medium
- **Found by:** Concurrency (C2)

### [I4] Negative game score renders as an unsigned 16-bit number ("Score : 65526")
- **File:** `src/translate/statusTranslate.ts` (`RIGHT` regex `-?\d+` + formatter). Root cause: `ifvms.js/src/zvm/io.js` reads score via `getUint16`.
- **Bug:** The interpreter reads the score unsigned, so a negative score (e.g. death −10) arrives as `65526` and is reformatted verbatim into the translated status bar. The `-?` alternative in the regex is dead for the live interpreter. The file comment acknowledges this but doesn't correct it.
- **Impact:** Visibly wrong score in the translated status bar on a foreseeable path (player death / penalties). English has the same defect, but the translation layer is the natural correction point.
- **Suggested fix:** Reinterpret the captured score as signed 16-bit before formatting (`v >= 0x8000 ? v - 0x10000 : v`). Ideally fix in `parseStatus`/the bridge so the English path is correct too.
- **Confidence:** Medium
- **Found by:** Logic (L2) AND Error Handling — agreement.

## Suggestions

- **[S1] Transient live-miss failures are terminal (no retry).** `src/translate/useOutputTranslation.ts` — a miss that settles to `'english'` (watchdog timeout from gate congestion, momentary unload, empty output) leaves `basisRef` set to `en`, so the scan guard skips that id forever. On a non-cache, non-empty *failure* settle, clear `basisRef` for that id (or mark `'failed'` with a bounded retry) so a later render re-attempts once the engine is free. (Concurrency C3, conf 62.)
- **[S2] Cache-poisoning defense-in-depth.** `src/translate/xlPrompt.ts` + `cacheSet` — a prompt-injected/garbage fallback translation persists to IndexedDB forever for that line (no XSS — renders as a plain React text node; verified `dangerouslySetInnerHTML`/`innerHTML` absent). Cap cached fallback length and/or version-tag the cache key so a corpus/prompt bump invalidates poisoned entries. (Security S1, accepted risk, conf 75.)
- **[S3] EngineGate input-priority can starve output under sustained typed-ahead.** `src/llm/engineGate.ts` — handoff always prefers the first `'input'` waiter with no aging; the output watchdog only starts *after* gate acquisition, so visible-pending time is unbounded in principle. Practically bounded (the F-A input queue is capped and drains one-at-a-time, per spec §6's deliberate input-first choice). Consider a "max queued age" cap if shimmer-forever is ever observed. (Concurrency C4, conf ~50.)
- **[S4] missLog re-parses up to 200 entries per miss.** `src/translate/missLog.ts` — bounded by `MISS_CAP=200` on the rare miss path (no DoS); an in-memory `Set` of seen `(game,language,en)` keys would avoid the re-parse. (Security S3, conf 60.)
