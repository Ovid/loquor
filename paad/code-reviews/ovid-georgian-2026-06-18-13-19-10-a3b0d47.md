## Executive Summary

This branch adds Phase-1 Georgian (`ka`) **output** translation: the game display
is translated to Georgian while the player still **types English** (raw-send; no
NL input layer for `ka`, and no LLM fallback — small models can't produce
Georgian). The implementation is unusually careful and well-documented: the
`lang`/`lexLang` split provably keeps `ka` out of the LLM fallback, the
`outputOnly`/`nlInputOn` derivation cleanly raw-sends English, modal/upgrade
suppression flows from a single source, and contract changes (`ActiveLanguage`
gaining `ka`, `LexLang` pinning, the new `sig` field) are handled correctly.
**No Critical or correctness-breaking issues were found.** The one issue worth
fixing before merge is an **a11y defect in an a11y feature**: the bilingual beta
notice voices its English half with Georgian pronunciation. The rest are
low-priority suggestions (doc staleness, a gate-coverage gap, string
duplication, a latent Phase-2 landmine, a missing test).

## Important Issues

### [I1] Bilingual beta notice mis-voices its English half (`lang="ka"` wraps English text)
- **File:** `src/ui/Terminal.tsx:175-178` (string) and `:266-270` (render)
- **Bug:** The beta notice is a single `<p className="nl-notice" lang="ka">`
  containing **both** the Georgian sentence and the English sentence "Georgian is
  a beta translation; some text may still appear in English." Because `lang="ka"`
  applies to the whole element, a screen reader switches to a Georgian voice and
  reads the English half with Georgian phonemes — garbled for the English-reading
  user that half exists to serve.
- **Impact:** This is on the main Phase-1 player path (Georgian output, Zork I),
  and the notice is itself an accessibility feature (spec §5: "told in their own
  language and English"). Per CLAUDE.md, a11y is a hard requirement; the rest of
  the codebase is meticulous about per-language `lang` attributes (nl-source
  lines, the `you` pill in `lang="en"`), so this notice breaks the project's own
  rule. Visually it's fine — only screen-reader pronunciation degrades.
- **Suggested fix:** Split into two segments with correct `lang`:
  `<p className="nl-notice"><span lang="ka">…</span>{' '}<span lang="en">…</span></p>`
  (two `<p>`s also work). Then add a test asserting the English half carries
  `lang="en"` (see S5).
- **Confidence:** High (88)
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Accessibility (3 agents agreed)

## Suggestions

- **[S1] `notices.ts` non-`ok` branch bypasses the `byLang` en-fallback.**
  `src/llm/notices.ts:112` returns `` `${prefix[lang]} — ${body[lang]}` ``,
  indexing the now-`Partial` maps directly instead of via `byLang(...) ?? m.en!`
  like every sibling. For a language with no entry (`ka`) this yields
  `"undefined — undefined"`. **Unreachable in Phase 1** (ka is output-only and
  raw-sends, never calling `nl.translate`), but a Phase-2 landmine and
  inconsistent with the file's own fallback contract. Fix:
  `` return `${byLang(prefix, lang)} — ${byLang(body, lang)}` ``. (Confirmed, latent — confidence 70)

- **[S2] "No English passthrough" gate covers only STRINGS, not OBJECTS/TEMPLATES.**
  `src/translate/corpus/index.test.ts:39` filters only `ZORK1_KA_STRINGS` for
  `en === ka`. Objects and templates are ungated. No leak today (current values
  verified Georgian), but a future English-identical object/template entry would
  render English-in-Georgian mid-transcript without failing any gate. Extend the
  gate to all three maps. (Confirmed — confidence 75)

- **[S3] CLAUDE.md is stale on output-translation languages.** The "Repository
  state" header and source-of-truth #2 describe output translation as
  "French/Spanish/German" / "EN/FR/DE/ES"; `ka` is now a 5th output language.
  Update when this lands. (Confirmed — confidence 85)

- **[S4] Georgian beta sentence is duplicated across two layers.** Hardcoded
  inline at `src/ui/Terminal.tsx:177` and as a reworded independent variant at
  `src/ui/landingStrings.ts:175-177` (`ka.caveat`). Two separate literals that
  can drift (e.g. a native-review wording fix applied to one only). Source the
  Terminal string from a strings table or cross-link them. Low impact — both are
  flagged "draft pending native review". (Confirmed — confidence 70)

- **[S5] Missing tests for the corpus-null `ka` path.** `Landing.test.tsx` never
  selects `ka`, so the "English only" badge on Zork II/III is untested.
  `Terminal.test.tsx:371-403` tests the Zork I beta notice (presence + that it
  lands in `role="status"`) but not the Zork II/III path (corpus null → display
  English, beta notice suppressed). That suppression is load-bearing — it's what
  prevents a misleading "beta translation" claim on untranslated volumes — so a
  future refactor could silently regress it. No test asserts the beta notice's
  `lang` split (the I1 bug ships unguarded). Add: ka + Zork II → no beta notice;
  ka selected → badge on untranslated volumes. (Partially confirmed — confidence 62)
