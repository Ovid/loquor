## Executive Summary

This branch implements **Georgian (`ka`) input — Phase 2**: a Georgian input
lexicon + parser pre-stage (`expandGeorgian`), a G1 dative path, Georgian
directions/conjunctions, signature-gated UI copy, and model/queue guards so `ka`
stays grammar-only. The implementation is substantial, well-tested (the suite is
green), and closely tracks its design/plan; the review-fix citations (C1, C2, M2,
G1) genuinely match the pushback docs. The dominant risk is structural, not
sloppy: **`ka` has no LLM fallback ("no LLM net"), so every deterministic gap is
a forced-English leak or a stuck/wrong path** — and the green walkthrough gate
only exercises one phrasing of each command, hiding natural variants. The
highest-severity finding is **[C1] a G1 dative path that silently emits the wrong
recipient** for common `give`-verb commands (confirmed at runtime). Three
Important findings follow (a stranded "take everything", a Phase-1/Phase-2
activation-tip race for returning players, and untranslated Georgian
disambiguation replies). All seven findings were confirmed against the code;
four were reproduced by running the real parser.

## Critical Issues

### [C1] G1 dative path emits the wrong recipient for any two-noun `give`/`tie` command whose second noun's stem ends in `ს`
- **File:** `src/llm/lexicon/parse.ts` (the block guarded by `core.postpositions && tokens.length === 2 && tokens[1].endsWith('ს') && verbArityOk(verb, vocab, 2)`)
- **Bug:** The dative-recipient path keys solely on `tokens[1].endsWith('ს')`. Georgian dative is `-ს`, but several **non-recipient** Zork I noun stems natively end in `ს`: `თას` (chalice), `სკარაბეუს` (scarab), `პლასტმას` (plastic), `სახრახნის` (screwdriver). The in-code comment claims the path is "bounded to the closed Zork I recipient set," but **no membership check exists**.
- **Impact:** A two-object command silently mistranslates to a wrong `X to Y`. Confirmed at runtime: `მიეცი თასი სკარაბეუსი` → `give chalice to scarab`; `მიეცი ბურთულა სახრახნისი` → `give bauble to screwdriver`. Legit datives (`მიეცი კვერცხი ქურდს` → `give egg to thief`, `მიაბი თოკი მოაჯირს` → `tie rope to railing`) still work, so the walkthrough gate stays green and hides it. With no LLM net, the player gets a confidently-wrong command, not an abstain.
- **Suggested fix:** Enforce the stated invariant — gate the path on membership in an explicit closed `KA_DATIVE_RECIPIENTS` set (the entries listed in `-ს` form, e.g. `ქურდს`, `მოაჯირს`), not a bare `.endsWith('ს')` suffix test. This is exactly what plan-pushback item M3 recommended (`paad/pushback-reviews/2026-06-24-loquor-georgian-input-plan-pushback.md:96-97`, which warned the broad check was "safe only by luck of the Zork I noun set"); runtime shows it is not even safe by luck.
- **Confidence:** High (reproduced at runtime)
- **Found by:** Logic & Correctness, Contract & Integration, Plan Alignment; verified

## Important Issues

### [I1] "take everything" / "drop everything" (`ყველაფერი`) silently misses and strands the player
- **File:** `src/llm/lexicon/expandGeorgian.ts` (the `token.endsWith('ი')` strip) + `src/llm/lexicon/parse.ts` (strip runs before the quantifier match) + `src/llm/lexicon/ka.core.ts` (`quantifiersAll`)
- **Bug:** `expandGeorgian` strips a trailing `-ი` from **every** object-span token before the quantifier branch runs. `quantifiersAll` contains `ყველაფერი` ("everything"), but the strip turns it into `ყველაფერ`, which is not in the set.
- **Impact:** Confirmed at runtime: `აიღე ყველაფერი` and `დადე ყველაფერი` → `{kind:'miss'}` → ka abstains with nothing sent (no LLM net). `აიღე ყველა` ("take all", vowel-final, strip is a no-op) works. "Take everything" is one of the most natural Zork commands; the design (§4.1) explicitly lists `ყველაფერი` as supported. The walkthrough gate only uses `ყველა`, so the gap is invisible to the suite.
- **Suggested fix:** Store the bare stem `ყველაფერ` in the quantifier set (and/or skip the `-ი` strip for quantifier tokens). A targeted test for `ყველაფერი` should accompany the fix.
- **Confidence:** High (reproduced at runtime)
- **Found by:** Logic & Correctness, Plan Alignment; verified

### [I2] Activation-tip race: a returning `ka` player on Zork I can be latched to the wrong Phase-1 "type in English" tip
- **File:** `src/llm/useNaturalLanguage.ts` (activation effect, ~lines 223-252) + `src/llm/notices.ts` (`makeActivationNotice` once-per-language `shown` latch) + `src/ui/useGameEngine.ts` (`signature` resolves async) + `src/llm/useModelDownload.ts` (boot-restore flips `internal` to `on/ka` independent of signature)
- **Bug:** The activation effect computes the tip via `kaInputActive(active, signature)` per fire, but `signature` starts `''` and resolves asynchronously, **independently** of the NL cache probe that boot-restores a stored `ka` preference. If `ka` activates while `signature === ''`, `kaInputActive('ka','')` is `false` → the effect announces **and latches** the Phase-1 `GEORGIAN_ACTIVATION_TIP_TYPE_ENGLISH`. When `signature` later resolves to Zork I, the effect re-runs but the guard `if (active === prevActiveLangRef.current) return` early-returns and the latch is spent → the correct Phase-2 tip never shows.
- **Impact:** A returning Georgian player (stored `ka` pref) on Zork I is told, in Georgian, to "type in English" — the opposite of the new capability — and never learns Georgian input works or about the quoted-escape hatch. Timing-dependent (likely on a warm cache), so intermittent and easy to miss in manual QA. Every activation test passes a static `signature`, so the ordering is untested.
- **Suggested fix:** Defer the `ka` activation notice until `signature !== ''` (mirror the existing modal-open deferral that leaves `prevActiveLangRef` unset so the effect re-runs), or key the latch on `(lang, kaInput)` so it can re-fire once when `kaInput` flips false→true for the same language.
- **Confidence:** High
- **Found by:** Logic & Correctness, Contract & Integration, Concurrency & State; verified

### [I3] Georgian disambiguation/orphan replies are raw-sent untranslated (breaks the natural dam-puzzle flow; escapable)
- **File:** `src/llm/translatePipeline.ts` (disambiguation/orphan prompt handling, ~lines 661-680 and the drain twin)
- **Bug:** For a disambiguation/orphan prompt (not a confirmation), the reply is `reply = line` (the raw player input) → `sendTracked(reply)`. The corpus renders the prompt **in Georgian** (e.g. the dam's 4 buttons → `push button`/`დააჭირე ღილაკი` yields a guaranteed 4-candidate prompt), so the player naturally answers with the Georgian word they see — which is raw-sent verbatim to the English Z-parser and never resolves. A bare-noun reply also can't go through `parseLexicon` (it requires a verb).
- **Impact:** The natural "answer the prompt" flow is broken for `ka` on a golden-path puzzle. **Escapable:** re-issuing a fully specified command works (`დააჭირე წითელი ღილაკი` → `push red button`), so the player is not permanently stuck, but a player who doesn't know to rephrase is. This is the `{raw}`-echo / no-LLM-fallback gap the disambiguation reframe (Task 22) addressed for *display* but not for the *reply* path.
- **Suggested fix:** Route a disambiguation/orphan reply for `ka` through a noun-only resolver (resolve the Georgian candidate word against the noun lexicon before `sendTracked`), or surface a Georgian hint that the player can answer with the quoted-English escape. Per CLAUDE.md's "player experience overrides product decisions" rule, this is worth a quick conversation if a non-trivial fix is needed.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases; verified

## Suggestions

- **[S1] `containsGeorgian` misclassifies all-Mtavruli input** — `src/llm/translatePipeline.ts` (`/[Ⴀ-ჿ]/`, ~line 76; used at the ka raw-send branch ~line 915). `line` is not lowercased before the test, so an all-Mtavruli (U+1C90–1CBF) line that *misses* the lexicon raw-sends Georgian bytes to the Z-parser (English error + burned turn) instead of abstaining. A lexicon *hit* is unaffected (fold normalizes first). Fix: `/[Ⴀ-ჿ]/.test(s.toLowerCase())`. (Reproduced: `re.test('Ა')` false, `re.test('Ა'.toLowerCase())` true.)
- **[S2] `expandGeorgian` mangles a bare `ში` to `შ`** — `src/llm/lexicon/expandGeorgian.ts`. A token equal to a 2-char postposition isn't split (`2 > 2` false) then loses its `-ი`. Low impact (a bare bound suffix is unrealistic input; a miss abstains with no English leak). Fix: skip the `-ი` strip when the token is itself an exact postposition, or require a minimum stem length.
- **[S3] BottomBar persistent tip is keyed on corpus presence, not `kaInputActive`** — `src/ui/BottomBar.tsx` (hardcodes the Phase-2 `GEORGIAN_ACTIVATION_TIP`) gated by `showBeta` (`outLang==='ka' && corpusFor(signature,'ka')!==null`) in `src/ui/Terminal.tsx`, while the one-shot announce selects Phase-1/Phase-2 via `kaInputActive`. The two predicates (output-corpus vs input-lexicon) coincide only because Zork I is the sole game with both; a future corpus-but-no-input game would show "type Georgian" while input raw-sends English — the drift spec Decision 6 forbids. Latent today. Fix: drive the BottomBar tip from `kaInputActive` too.
