## Important Issues

### [I1] `distributePrepTail` distributes a source preposition, producing a command the player never typed
- **File:** `src/llm/inputTranslate.ts:232-251` (prep detection at `:198-215`; emit gate at `src/llm/lexicon/parse.ts`)
- **Bug:** The function appends the last clause's trailing `<prep> <indirect>` to all preceding same-verb conjuncts on the assumption that the tail is a *shared destination* ("lege A und B **in die Vitrine**"). But `aus`/`von`/`unter` are real core prepositions (`de.core.ts` preps), so a *source* tail is detected and distributed too. `"nimm das schwert und nimm den schlĂĽssel aus der vitrine"` (take the sword; take the key **from** the case) â†’ first clause becomes `"nimm das schwert aus der vitrine"`.
- **Impact:** `take` is in `verbs2`, so the rewritten clause does NOT miss to the LLM â€” it emits the real command `take sword from case`. The player's plain "take the sword" is silently executed as "take sword **from** the case." Wrong-emit on natural phrasing â€” the exact player-harm class CLAUDE.md calls out. Frequency is low (source-prep takes are rarer than put-in-container), but it is a wrong action, not a no-op.
- **Suggested fix:** Gate distribution on the tail's preposition being a destination/locative sense (`in`/`on`/`under`/`to`); skip when it resolves to `from` (`aus`/`von`). Simplest implementation: check `core.preps[fold(tailPrepToken)]` against a destination allowlist before inheriting. The endgame "put treasures in case" loop the feature targets is entirely `in`/`on`, so narrowing keeps the intended behavior.
- **Confidence:** High (85)
- **Found by:** Logic & Correctness; confirmed + upgraded by Verifier (wrong-emit, not degrade-to-LLM)

### [I3] German player's reflex "j"/"ja" is not accepted on yes/no confirmation prompts (player-experience â€” needs a conversation)
- **File:** `src/llm/inputTranslate.ts:361-378` (`CONFIRM_PROMPT`/`isConfirmationPrompt`); corpus `src/translate/corpus/zork1.de.strings.ts:1138,1232`
- **Bug:** The confirmation prompt is displayed in German as "(Y bedeutet ja)" / "(Y fĂĽr ja)", but it keeps the literal key "Y". A German speaker's reflex affirmative is "j" (or "ja"). "j" is neither a meta command nor a vocab word, and nothing maps it to "y", so on a restart/quit/restore-overwrite prompt it is passed raw to the interpreter, which only accepts "Y" â€” the confirmation silently fails.
- **Impact:** A natural keystroke fails on a confirmation, with no feedback the player can diagnose. Adjacent to the F2 fix this branch just made.
- **Suggested fix:** When `isConfirmationPrompt` is active, map a German affirmative ("j"/"ja"â†’"y", "n"/"nein"â†’"no") before sending raw, and/or adjust the displayed disclosure. **Raise with Ovid â€” display/product choice.**
- **Confidence:** Medium (75)
- **Found by:** Error Handling & Edge Cases; Verifier
