## Important Issues

### [I1] Entering the game silently clobbers a saved `off` (NL-disabled) preference

- **File:** `src/ui/Landing.tsx:61-64` (read) + `src/ui/Landing.tsx:183-186` (write); downstream `src/llm/useModelDownload.ts:97-131, 269-272`
- **Bug:** The picker's `language` state is initialized from `readNlPref().language` with a documented `off ŌåÆ en` display mapping (`Landing.tsx:61-64`). The "Light the lamp ŌåÆ" handler then calls `writeNlPref({ language })` **unconditionally** ŌĆö there is no "did the player actually touch the picker" dirty flag (`Landing.tsx:183-186`). So a saved `off` is read as `'en'` and written back as `'en'`, destroying the `off` on every landingŌåÆenter round-trip (including a plain page reload, and via the "Change story" overlay, which remounts fresh each open and re-runs the same handler).
- **Impact:** `off` is the exact mechanism by which an in-game "Off" choice persists across reloads (`useModelDownload.ts:269-272` writes `language:'off'`; the boot probe at `useModelDownload.ts:97-131` only restores `phase:'on'` when `pref.language !== 'off'`). Overwriting it to `en` silently **re-enables a layer the player explicitly turned off**. Confirmed NEW on this branch ŌĆö `main`'s `Landing.tsx` has no language state and never writes the pref. No test guards it: `Landing.test.tsx` asserts the `offŌåÆen` _display_ mapping but never that a saved `off` survives entry. (F2 and F4 from the review are the same root cause seen from the overlay-re-entry and dual-ownership angles: the `writeNlPref` runs _before_ `App.enter`'s same-story early-return, and the in-game picker and overlay Landing both own `nlPref.language`.)
- **Suggested fix:** Only persist when the player actually changed the picker in this Landing session (track a `touched` flag; `if (touched) writeNlPref({ language })`), and never map a saved `off` to `en` for persistence ŌĆö preserve `off` round-trip. Add a regression test asserting a saved `off` survives landingŌåÆenter.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling & Edge Cases, Concurrency & State (verifier: Confirmed, live)
- **Note (CLAUDE.md "talk to me first"):** silently re-enabling the NL layer for a player who disabled it is a player-facing behavior change ŌĆö worth a decision from Ovid, not a silent fix.

## Suggestions

- **[S1] `LanguageCombobox` is fragile on an empty / shrinking `options` list** ŌĆö `src/ui/LanguageCombobox.tsx:49,93,110,116`: `current = options.find(...) ?? options[0]` is `undefined` if `options` is `[]` (then `current.lang`/`.label` throw), and `options[active].value` throws if `active` outruns a shrunk list. Both call sites pass fixed module constants (`LANGUAGE_OPTIONS` = 5, `LANDING_LANGUAGES` = 4), so it is **latent/defensive**, not a live bug ŌĆö but the component is now advertised as shared/caller-owns-the-list. Add `if (options.length === 0) return null` and clamp `active` with `Math.min(active, options.length - 1)`. (Logic, Error Handling, Concurrency.)
- **[S2] Caveat copy may read as "my language might not work" (player-experience)** ŌĆö `src/ui/Landing.tsx:148-152`: "ŌĆ”an optional, experimental model ŌĆö a one-time download that **may not support every language**." - **[S3] `aria-controls` references a listbox id that only exists while open** ŌĆö `src/ui/LanguageCombobox.tsx:108` vs `:121-125`.
