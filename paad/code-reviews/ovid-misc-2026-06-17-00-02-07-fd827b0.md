upgrade; picking a language activates the deterministic pipeline immediately) plus

## Critical Issues

### [C1] `setLanguage('off')` mid-download silently re-enables NL and overwrites the player's explicit Off

- **File:** `src/llm/useModelDownload.ts:249-252` (with `:187-203`)
- **Bug:** When the player chooses **Off** while a model download is in flight,
  `setLanguage('off')` sets `{phase:'off'}` and persists `off`, but does **not**
  `abortRef.current?.abort()` and does **not** clear the stall timer. The in-flight
  `engine.load(...).then(...)` checks only `stale()` =
  `ac.signal.aborted || abortRef.current !== ac` — neither is true after a plain
  `setInternal({phase:'off'})`. So when the load resolves it runs
  `setInternal({phase:'on', model:'full'})` + `writeNlPref({language:pendingLangRef})`,
  re-enabling the NL layer and **persisting** a language the player just turned off.
  The stall timer can likewise fire and flip to `on/grammar`.
- **Impact:** A player who explicitly disables the NL layer mid-download has it
  silently switched back on — and that choice persisted across reloads. This is
  state corruption against an explicit user action, exactly the class the pipeline
  side guards against ("'off' must restore raw play instantly"). The existing "off"
  test only covers off-from-cached, so this path is untested.
- **Suggested fix:** In `setLanguage('off')` (and arguably any language switch),
  `abortRef.current?.abort()` and clear the stall timer before setting phase.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases (verified)

## Important Issues

### [I1] Cached-model pick during the async boot probe → grammar-only + spurious download modal

- **File:** `src/llm/useModelDownload.ts:85-111` and `:255-263`
- **Bug:** The boot probe (`engine.isCached().then(...)`) only restores the persisted
  language while `prev.phase === 'off'`. If the player opens the picker and chooses a
  language **before** `isCached()` resolves, `installed` is still `false` and
  `engine.isLoaded()` is `false` (cached on disk, not in VRAM), so `setLanguage`
  commits `on/grammar` and opens the upgrade modal. The probe then resolves to a
  no-op (`prev.phase` is now `'on'`).
- **Impact:** A player whose model **is** cached is dropped into grammar-only and
  shown the large-download upgrade modal — contradicting the spec's "cached →
  straight to on/full, no modal" (design lines ~63/89-90/209). On a warm reload this
  is a common path and the modal is the most visible regression.
- **Suggested fix:** In `setLanguage`, when not yet `installed`, consult
  `engine.isCached()` before committing to grammar/opening the modal; or have the
  probe re-correct an `on/grammar` phase up to `full` when it finds the model cached
  (let the probe upgrade, not just restore-from-off).
- **Confidence:** Medium-High
- **Found by:** Concurrency & State (verified)

### [I2] Duplicated, a11y-critical focus-trap logic in the modal and the landing plate

- **File:** `src/ui/ModelDownloadModal.tsx:89-126` duplicates `src/ui/Landing.tsx:39-54`
- **Bug:** The modal's focus trap was newly added on this branch and is a verbatim
  copy of Landing's: same `FOCUSABLE` selector string, the same Tab/Shift-Tab
  first↔last cycle, the same Esc-to-dismiss + focus-restore-on-close. No shared
  helper exists (`grep` for `useFocusTrap`/`trapFocus` finds nothing).
- **Impact:** CLAUDE.md makes a11y a hard requirement. Two independent copies of
  focus-trap code drift — an a11y fix to one silently leaves the other broken.
- **Suggested fix:** Extract a `useFocusTrap(ref, { onEscape })` hook (or a
  `<FocusTrap>` wrapper) and use it in both.
- **Confidence:** High
- **Found by:** Contract & Integration (verified)

### [I3] English `downloading` text shown verbatim in the otherwise-localized upgrade modal

- **File:** `src/ui/Terminal.tsx:139` (`text: 'downloading'`) → rendered at `src/ui/ModelDownloadModal.tsx:149`
- **Bug:** Every other string in the reframed modal is localized via `COPY[lang]`
  (EN/FR/DE/ES), but the dynamic progress line renders the hardcoded English
  `progress.text`, so a French/German/Spanish player sees `"45% — downloading"`.
- **Impact:** Mixed-language artifact in exactly the localized upgrade flow this
  branch built; violates the "a fix in one language is a fix in all" rule. Pre-existing
  on `main`, but this branch made it visibly inconsistent by localizing everything around it.
- **Suggested fix:** Drop `progress.text` from the rendered string (the localized
  `progressLabel` + `pct%` already convey state), or add a localized "downloading"
  to `ModalCopy`.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases + Contract & Integration (agreed; verified)

### [I4] ETA string is English-only inside the localized modal

- **File:** `src/llm/progress.ts:45-55` (`formatEta`) → rendered at `src/ui/ModelDownloadModal.tsx:99,150`
- **Bug:** `formatEta` only ever returns English (`"~10s remaining"`, `"~3 min remaining"`,
  `"~1h 5m remaining"`), so the localized modal shows two English fragments
  (`"… — downloading · ~3 min remaining"`) for FR/DE/ES players.
- **Impact:** Same i18n leak as I3, same rule. The test hard-asserts the English form,
  confirming there's no localized path.
- **Suggested fix:** Pass `lang` into `formatEta` (or a thin localized wrapper) and
  translate the unit words + "remaining"/"min" across EN/FR/DE/ES.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases (verified)

### [I5] "…thinking" status is hardcoded English, untagged, shown to non-English players

- **File:** `src/ui/Terminal.tsx:183` (`<p className="nl-thinking">…thinking</p>`)
- **Bug:** Shown to a FR/DE/ES player mid-translation while the sibling queued lines
  and notices in this same diff were carefully given `lang={nlLang}`. Pre-existing on
  `main`, but it's the exact "non-English player sees English UI" gap the rest of the
  change set closes.
- **Impact:** Untranslated status text + no `lang` attribute (WCAG 3.1.2). Same
  cross-language rule as I3/I4.
- **Suggested fix:** Localize via `notices.ts` and tag with `nlLang`.
- **Confidence:** Medium
- **Found by:** Logic & Correctness (verified)
