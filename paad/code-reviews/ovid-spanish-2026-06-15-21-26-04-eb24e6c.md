## Important Issues

### [I1] `loudEcho` uses the whole raw line's last word — compound/multi-word commands mis-echo

- **File:** `src/translate/loudEcho.ts:32-34`, `src/ui/Terminal.tsx:148-153`
- **Bug:** `setLastInput(text)` stores the entire raw typed line; `loudEcho` keeps only its last whitespace token. For a compound like `coge la barra y mira`, the VM echoes the _first_ clause's action but `loudEcho` injects `mira` → `mira mira ...`.
- **Impact:** Wrong-word echo on a legitimate, NL-supported compound input. The file header acknowledges this as an "accepted heuristic limit," but it is a genuine defect, not a non-issue.
- **Suggested fix:** Substitute the per-clause command actually sent to the VM (the pipeline knows it) rather than `lastInput`'s whole-line last word; or only fire `loudEcho` for bare single-clause commands.
- **Confidence:** Medium-High
- **Found by:** Error Handling, Concurrency & State (2 specialists)

### [I2] Stale `lastInput` retroactively restamps historical echo lines in scrollback

- **File:** `src/translate/useOutputTranslation.ts:264-290` (memo + dep array at :290)
- **Bug:** The `lines` useMemo re-maps **every** `view.lines` entry on each render and lists `lastInput` in its deps. An older Loud Room echo line still in scrollback is re-rendered with the _newest_ command's last word whenever `lastInput` advances. Echoes are not pinned to the turn/line that produced them.
- **Impact:** Already-rendered transcript history visibly mutates as the player keeps playing (and on repeat Loud Room visits). Race variant: with NL queuing, an echo emitted on a later async render can receive a newer command's word (Concurrency specialist rated this the highest-confidence symptom at 85).
- **Suggested fix:** Resolve the echo word once when the line first renders into scrollback and freeze it onto the line (store substituted text), or key the echo to the turn that emitted it rather than to a single global `lastInput`.
- **Confidence:** Medium-High
- **Found by:** Logic & Correctness, Contract & Integration, Concurrency & State (3 specialists)

### [I3] `loudEcho` has no Loud-Room location gate — the regex hijacks any matching line

- **File:** `src/translate/loudEcho.ts:20,30`; consumed at `src/translate/useOutputTranslation.ts:206,278`
- **Bug:** `loudEcho` fires purely on the text shape `/^(\S+) \1 \.\.\.$/`, with no check that the player is actually in the Loud Room. `view.status.location` is available (`StatusLine.location`) and threaded into the hook, so gating is feasible but not done. Any line (this game or a future one) matching the doubled-word-ellipsis shape gets overwritten with the player's last typed word — and this path runs _before_, and takes precedence over, the corpus/LLM overlay and suppresses miss-logging.
- **Impact:** Unrecoverable false-positive substitution on any coincidental doubled-word line. The "no other Zork I line collides with it" comment is asserted, not enforced.
- **Suggested fix:** Gate `loudEcho` on `view.status?.location` being the Loud Room before substituting. This also shrinks the I1/I2 race window to Loud-Room turns only.
- **Confidence:** Medium
- **Found by:** Logic & Correctness, Concurrency & State (2 specialists)

### [I4] `zork1.es.templates.ts` header is stale and contradicts the implementation

- **File:** `src/translate/corpus/zork1.es.templates.ts:6-10`
- **Bug:** The header asserts _"NEVER put «de» or «a» immediately before a slot… pinned per-object as a full string."_ But the ES implementation added `withContractions()` (`zork1.es.objects.ts:443-458`) deriving `alDef`/`delDef` forms, and ES templates **do** place de/a before slots via them (e.g. `:195` `No hay nada detrás {obj.delDef}`, `:199` `{obj.alDef}`, `:70`, `:77`). The header was copied from the French corpus (which genuinely follows the rule) and never updated.
- **Impact:** Misleading documentation in exactly the file whose contract it describes; future edits may follow the stated (wrong) rule.
- **Suggested fix:** Rewrite the ES header to document the `alDef`/`delDef` contraction-form mechanism.
- **Confidence:** Medium
- **Found by:** Plan Alignment (1 specialist)

## Suggestions

- **[S1] `loudEcho` strips `«»`/trailing punctuation but not leading `¡`/`¿`** (`src/translate/loudEcho.ts:35-37`). `fold.ts:22` strips them; a player typing `¡mira!` echoes `¡mira ¡mira ...`. Fix: `.replace(/^[¡¿"'«»]+/, '')`. Cosmetic, but in the language this branch targets.
- **[S2] `lastInput` in the heavy miss-logging effect's dep array** (`src/translate/useOutputTranslation.ts:262`) forces a full `view.lines` re-scan on every submit even when `view` is unchanged. Verified idempotent (basis/backlog/status guards) — pure wasted work, not a correctness bug. Fix: read `lastInput` via a ref inside the effect; keep it as a dep only in the `lines` memo.
- **[S3] `capitalizeFirstLetter` is not surrogate-pair safe** (`src/translate/match.ts:34-35`): `charAt` on a `\p{L}` code-unit index splits an astral-plane initial letter. Latent only — es/fr corpora are BMP-only. Fix: `String.fromCodePoint(s.codePointAt(i))`.
- **[S4] Scope deviation (process note, not a code bug):** spec §1 permits only `KNOWN_COLLISIONS.es` as an input-layer edit, but the branch modifies `src/llm/inputTranslate.ts` and `src/llm/translatePipeline.ts`. The changes are tested and sound; flagging the spec/scope mismatch for governance.
