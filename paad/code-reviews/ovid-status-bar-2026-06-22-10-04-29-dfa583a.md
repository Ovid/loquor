## Executive Summary

This branch renames the Georgian-only footer (`GeorgianStatusBar`) to an
always-present `BottomBar` rendered in every language, adds a pure
`nlModeSummary()` helper, and a diagnostic readout (NL-mode + story title +
debug-gated signature). The code logic is clean — no correctness, contract,
null-path, or edge-case bugs were found, and the union switch is
typecheck-enforced exhaustive. The findings are concentrated on **one design
decision**: making the bar unconditional contradicts the approved spec's
Decision 5 (`ka`-only) without a companion design doc, which cascades into a
real a11y gap (missing `lang="en"`), three stale comments, a permanent vertical
layout cost, and a player-experience question that CLAUDE.md says needs a
conversation. Highest-severity item: the always-visible English readout lacks
`lang="en"` (WCAG 3.1.2).

## Important Issues

### [I1] Always-visible English readout has no `lang="en"` (WCAG 3.1.2)
- **File:** `src/ui/BottomBar.tsx:43-46`
- **Bug:** The `.bottombar-readout` span renders `nlModeSummary(nlState)` —
  hardcoded English diagnostic tokens (`off`, `downloading…`, `no NL`,
  `output-only`, `de · full · input`) — plus the English `storyTitle`, with **no
  `lang` attribute**. The document/UI language is `fr`/`de`/`es`/`ka` when those
  are active, so a screen reader voices this English text with the wrong
  language's phonemes; for `ka` it attempts Latin-script English as Georgian.
- **Impact:** WCAG 2.2 AA, SC 3.1.2 (Language of Parts) violation, on text now
  shown to **every** non-English player permanently. The rest of this very file
  is scrupulous about tagging (`lang="ka"` / `lang="en"` on the notices at
  `BottomBar.tsx:55,62,71,75`); the readout is the lone untagged exception. This
  is a *new* exposure — pre-branch the bar was `ka`-only and carried only
  Georgian-tagged content.
- **Suggested fix:** Wrap the readout span: `<span className="bottombar-readout"
  lang="en">…`. The content genuinely is English. Add a test asserting the span
  carries `lang="en"`, matching the existing `lang`-assertion pattern in the
  suite.
- **Confidence:** High (85)
- **Found by:** Accessibility & Responsive, Plan Alignment, Verifier

### [I2] Always-present readout ships diagnostic English chrome to all players — scope/PX deviation from the approved design
- **File:** `src/ui/Terminal.tsx:377-384`, `src/ui/BottomBar.tsx:43`, `src/ui/nlModeSummary.ts`
- **Bug:** `BottomBar` is rendered **unconditionally**. The approved design
  (`2026-06-21-…-design.md`, Decision 5, lines 78-80) locks the bar as `ka`-only
  ("absent (not empty) for en/fr/de/es … no layout change for those
  languages"), and its "Out of scope" section (line 188) explicitly lists "A
  per-language 'basic mode' line in the bar … **not built now**." There is **no
  companion 2026-06-22 design doc** (`ls docs/superpowers/specs | grep 2026-06-2`
  returns only the Georgian-only spec). The readout's own docstring
  (`nlModeSummary.ts:5-7`) calls it "diagnostic … **not player-facing copy**" —
  yet it is now permanent player-facing chrome.
- **Impact:** Every non-English player permanently sees untranslated English
  jargon (`fr · full · input — Zork I`, `off`, `downloading…`, `no NL`); a
  read-Georgian `ka` player sees `ka · output-only` in Latin script on an
  otherwise all-Georgian screen — reintroducing exactly the "untranslated
  English clutter" the 2026-06-21 work set out to remove. This falls under
  CLAUDE.md's "player experience / talk to me first" rule and the deviation is
  undocumented at the design level.
- **Suggested fix:** Talk to Ovid. Either (a) gate the readout behind `debug`
  (keep the always-present bar only for the `ka` notices), restoring the design's
  intent and the reclaimed layout space; or (b) keep it player-facing but
  localize it via the existing `notices.ts` by-language pattern AND write a
  companion design doc updating Decision 5 / the out-of-scope line.
- **Confidence:** High (80)
- **Found by:** Contract & Integration, Accessibility & Responsive, Plan Alignment, Verifier

### [I3] Stale comments contradict the new always-present behavior
- **File:** `src/ui/components.css:112-118`, `src/ui/nlModeSummary.ts:5-9`, `src/ui/Terminal.tsx:352-357`
- **Bug:** Three comments still describe the old `ka`-only arrangement:
  - `components.css:112-118` — section still titled "(ka only)" and claims the
    bar is "absent (not empty) otherwise, so `<main>` (flex:1) reclaims the space
    and there is no layout change." False — the bar always renders, so `<main>`
    never reclaims that space in any language.
  - `nlModeSummary.ts:5-9` — "for the debug strip in BottomBar … not player-facing
    copy." The summary is now always rendered (only the signature is debug-gated).
  - `Terminal.tsx:352-353` — "Both are ka-only, so en/fr/de/es get no extra DOM
    and `<main>` (flex:1) keeps the full height." The sibling `BottomBar` now
    always renders, so the "no extra DOM / full height" claim is wrong (the
    announce region the comment sits on *is* still ka-gated, but the surrounding
    claim is not).
- **Impact:** A future maintainer trusting these comments may re-add a render
  gate (reintroducing the empty-strip the team deliberately moved away from) or
  mis-style the bar as `ka`-only chrome. CLAUDE.md flags steering/comment
  contradictions explicitly.
- **Suggested fix:** Retitle the CSS block ("Bottom status bar — always
  present"), rewrite the "absent otherwise / no layout change" body, correct the
  `nlModeSummary` docstring ("the always-visible readout"), and scope the
  Terminal comment to just the still-ka-gated announce region.
- **Confidence:** High (90)
- **Found by:** Logic & Correctness, Contract & Integration, Error Handling & Edge Cases, Accessibility & Responsive, Verifier
