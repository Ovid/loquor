## Important Issues

### [I1] Orphan-prompt stop is missing from the turn-entry raw-reply guard

- **File:** `src/llm/translatePipeline.ts:477-480`
- **Bug:** The mid-sequence compound loop was updated to stop on `isConfirmationPrompt || isDisambiguationPrompt || isOrphanPrompt` (lines ~634-636), but the **turn-entry** guard that sends a reply RAW to the interpreter still checks only confirmation + disambiguation. For a _single_ command that produces a parser orphan prompt ("What do you want to put the coffin in?"), the player's next line answering it arrives as a fresh `translate()` call whose `recentOutput` is the orphan prompt. It falls through the guard and is split/translated instead of sent raw — in FR/DE/ES it abstains and sends nothing, so the orphaned command can never be completed.
- **Impact:** Non-English players cannot answer a parser orphan prompt after a single command; the action stalls silently. The compound-loop fix only covers the multi-clause case.
- **Suggested fix:** Add `|| isOrphanPrompt(recentOutput)` to the turn-entry guard at ~line 478, mirroring the mid-sequence check, and let the queued reply flush raw as the sibling prompts do.
- **Confidence:** High (88)
- **Found by:** Error Handling & Edge Cases (verified)

### [I2] `vocabVerbHeads` omits `verbSynonyms`, so verbSynonym-led compounds don't verb-gap

- **File:** `src/llm/inputTranslate.ts:53-63` (vs `src/llm/lexicon/parse.ts:138` `verbArityOk`)
- **Bug:** `vocabVerbHeads` builds leading-verb heads from only `verbsOnly + verbs1 + verbs2`, omitting `vocab.verbSynonyms`. But `verbArityOk` (and the FR `ulysse→ulysses` fix) treat `verbSynonyms` as real verbs. So in English mode (`core===null`) a verbSynonym-led compound — e.g. `"break the window and the door"` → `["break the window","the door"]` — never sets `lastVerb`, and the article-led conjunct `"the door"` is handed verbless to the LLM instead of gapping to `"break the door"`. This is exactly the wrong-verb-invention failure the gapping feature exists to prevent, re-opened for the verbSynonym verb class.
- **Impact:** Inconsistent gapping vs resolution: `parse.ts` recognizes a verb class the gap heuristic misses. Silent and language-dependent.
- **Suggested fix:** Include `...vocab.verbSynonyms` in the `vocabVerbHeads` set so the gap side matches `verbArityOk`'s notion of a verb.
- **Confidence:** High (80)
- **Found by:** Contract & Integration + Concurrency & State (verified)

## Suggestions

- **[S1]** `src/llm/inputTranslate.ts:31-34` — CLAUSE_SEP's Oxford-comma absorption `(?:(?:CONJ)\s+)?` only absorbs a conjunction followed by `\s+`; a _trailing_ conjunction ("go up, and", "take a, et") survives as a dangling bare-conjunction clause that reaches `parseLexicon` as a miss. Fix: allow end-of-string, e.g. `(?:(?:CONJ)(?:\s+|$))?`. (Logic & Correctness, conf 78)
- **[S2]** `src/llm/inputTranslate.ts` splitClauses / unquote contract — a partially-quoted line (`say "hello, world"`) is split mid-quote because `unquote()` only protects _fully_-quoted lines. Narrow (verbs taking punctuated quoted args are rare in Zork); mask quoted spans before splitting or document the limitation. (Logic & Correctness, conf 72)
- **[S3]** `src/llm/lexicon/de.core.ts` / `es.core.ts` — `quantifiersAll` was added only to `fr.core.ts`; DE/ES bare-"all" ("nimm alles" / "coge todo") still falls to the LLM (original bug persists for those languages). Port the field or note the deferral. Coverage gap, not a regression. (Contract & Integration, conf 85)
- **[S4]** `src/llm/lexicon/parse.ts:220-223` — the "all" branch emits `${verb} all` for any verb passing `verbArity1or2`, with no take/drop guard, so `lis tout`→`read all`, `ouvre tout`→`open all`. Mostly benign (ALL is a valid Z-parser object) but consider restricting to the take/drop family. (Logic + Error Handling, conf 60)
- **[S5]** Duplicated logic that may drift: `vocabVerbHeads` overlaps `vocabWordSet`; `startsWithArticle` re-hardcodes `['the','a','an']` (also hardcoded in `vocabWordSet`/`isVocabPassthrough`); `leadingVerbPhrase` re-implements `parseLexicon`'s idiom verb-extraction with a _different_ idiom sort key (word-count vs char-length). No behavioral divergence today, but the two sort keys and two verb-head sources should be unified. Note: I2 is a concrete instance of this drift. (Contract & Integration, conf 62)
- **[S6]** `src/llm/inputTranslate.ts:1` — stale path comment `// src/llm/translate.ts`; file is `inputTranslate.ts` (rename in commit d4b2850 left the header). (Concurrency & State, conf 95)
