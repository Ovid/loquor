## Executive Summary

The branch does two clean, well-understood things: (1) swaps the default UI theme
from dark to light (CSS selector flip + `useTheme` no longer auto-persists on
mount), and (2) kills a natural-language "language flash" by seeding the active
language synchronously on mount and adopting a language picked in the "Change
story" overlay. The theme swap is **fully self-contained and correct** — only
`theme.css` and `useTheme.ts` touch `data-theme`, nothing else reads the theme
value, the dark block redefines every variable the light `:root` sets (no
inheritance leak), and the cascade resolves correctly. The NL flash-fix logic is
sound (the falling-edge ref guard fires exactly once, the sync seed promotes
grammar→full via the existing boot probe).

**One real, verified defect:** the sync-seed timing change broke a sibling NL test
(`STALLED lazy engine.load demotes…`) that wasn't updated alongside the two that
were — it fails **deterministically in isolation** and survives the full suite
only by cross-test timing luck. Plus one **product/migration decision worth your
call** (Ovid): the dark→light default reaches **no pre-existing user**, because
the old code auto-froze `dark` into their storage. The rest are low-confidence
latent hardening notes.

## Important Issues

### [I1] Sync language seed broke the `STALLED lazy engine.load demotes` test — fails deterministically in isolation
- **File:** `src/llm/useNaturalLanguage.test.tsx:879-922` (root cause: `src/llm/useModelDownload.ts:76-81`)
- **Bug:** The new synchronous `useState` seed makes `internal.phase === 'on'`
  immediately in `model:'grammar'`. This test gates on
  `await waitFor(() => state.phase === 'on')` (line 901) and *then* calls
  `translate('sing a ditty')`, expecting it to run in **full** mode so the lazy
  `engine.load` stalls, the watchdog fires, and a "staying in basic mode /
  download failed" notice appears. On `main`, `phase` only became `'on'` *after*
  the boot probe promoted the cached language to `model:'full'`, so `waitFor`
  implicitly waited for that. With the seed, `waitFor` resolves on the **first**
  render while still `grammar` → `translate` raw-sends in grammar mode → no
  `engine.load`, no stall, no notice → `notice` is `null` → `.toMatch()` throws
  `"expects to receive a string, but got object"`.
- **Impact:** Verified: fails **3/3 in isolation**
  (`npx vitest run src/llm/useNaturalLanguage.test.tsx -t "STALLED lazy engine.load demotes"`)
  and failed on a full-file-alone run too; passes in the full suite only by
  accident of cross-test timing. Any run that isolates this file/test — CI
  sharding, `--isolate`, a focused run, or a future reorder — goes red. It also
  signals the seed's boot-timing contract change was propagated to two sibling
  tests (the decline test and the new seed test) but **not** this one.
- **Suggested fix:** Gate on the promotion the test actually depends on, not the
  bare phase: `await waitFor(() => { const s = hook.result.current.state; expect(s.phase === 'on' && s.model === 'full').toBe(true) })` before calling
  `translate`. (Same shape the other cached-restore tests use.)
- **Confidence:** High (reproduced directly)
- **Found by:** Contract & Integration

### [I2] Product/migration decision — the dark→light default reaches no pre-existing user (your call, Ovid)
- **File:** `src/ui/useTheme.ts:14` (new init reader) + the *removed* on-mount
  `localStorage.setItem` (old `main` effect persisted on every mount).
- **Issue:** Under the old code the default was `dark` **and** the effect wrote
  the theme to storage on every mount. So anyone who opened the app even once
  under old code — without ever toggling — already has `loquor-theme: 'dark'`
  frozen in storage. The new reader (`=== 'dark' ? 'dark' : 'light'`) reads that
  back as `dark`. Net: only brand-new installs (empty storage) see the new light
  default; **every returning user stays on dark**. There is no migration of the
  frozen value (and none is cleanly possible — an auto-frozen `dark` is
  indistinguishable from a deliberately-chosen `dark`, so clearing it would also
  reset genuine dark-choosers).
- **Why it's a "talk to me" item, not a code bug:** The mechanism is correct and
  the fix (persist only on toggle) is the right *forward* design. The harm is
  mild — case (b) users keep seeing exactly what they always saw, nobody is
  jarred mid-session — but the branch's headline behavioral change is effectively
  invisible to the existing user base. Per CLAUDE.md, a player-experience
  asymmetry like this should be a conscious decision, not an accident.
- **Walkthrough:** (a) new user, no value → **light** ✓; (b) old user, never
  toggled → frozen `dark` → **dark, never chosen** ⚠; (c) old user toggled light
  → `light` ✓; (d) storage blocked → **light** ✓.
- **Options:** (i) accept and document "the swap only affects new users" (least
  harmful — existing users keep their experience); (ii) a one-time clear of the
  frozen default — **not recommended**, it can't distinguish auto-frozen from
  deliberate dark.
- **Confidence:** High on the mechanism; the "is it harm / what to do" is yours.
- **Found by:** Error Handling & Edge Cases

## Suggestions

- **[S1]** `src/ui/Terminal.tsx:247-253` — the adoption effect's `chosen !== outLang`
  comparison conflates `'off'` with the `'downloading'` transient: during a
  download `outLang` is `'off'` (Terminal:218) while `readNlPref().language` is
  the real downloading language, so a falling edge then would `setLanguage(real)`
  → `abortInFlight()` and silently kill the in-flight multi-GB download.
  **Currently unreachable** — during a download `downloadingModalOpen` keeps
  `bgInert` true (Terminal:394→396→402), so the StatusBar "Change story" button
  is `inert` and can't be opened. Latent trap if that modal-gating ever changes;
  guarding the effect on `nl.state.phase === 'on' || 'off'` (skip `downloading`)
  would harden it. (Flagged by both Concurrency and Contract; they disagreed on
  reachability — I confirmed it is **not** reachable today.)
- **[S2]** `src/ui/useTheme.ts:30-38` — `toggle` derives `next` from the
  closed-over `theme` instead of a functional updater. Theoretical stale-closure
  lost-update under rapid/synthetic double-fire; practical likelihood ~zero
  (React flushes between discrete clicks). The functional form
  (`setTheme(t => …)` with the storage write inside, deps `[]`) is strictly safer
  at no cost and is what the code did before.
- **[S3]** `src/ui/Terminal.tsx:247-253` — the effect advances `prevInertRef`
  (consuming the falling edge) *before* `nl.setLanguage`, which early-returns if
  `!hasVocab`. If vocab weren't loaded at that instant the language pick would be
  silently dropped with no retry. Currently safe (vocab is a synchronous lookup
  and present whenever the in-game overlay is reachable); latent fragility if
  vocab ever loads async.
- **[S4]** `src/llm/useModelDownload.ts:137-146` — with the sync seed, the boot
  probe's `if (prev.phase === 'off')` activation branch is now dead for the
  stored-language case (the seed already made it `'on'`); promotion runs solely
  through the second branch (153-159). Two places now seed language and must stay
  in sync. Drop the dead branch or comment it as a defensive fallback.
