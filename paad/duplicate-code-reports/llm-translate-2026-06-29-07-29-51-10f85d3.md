## Executive Summary

This subsystem is **remarkably free of semantic duplication.** A 5-specialist hunt plus an adversarial verifier confirmed exactly **one** true semantic duplicate: a byte-for-byte identical regex-escape helper defined twice (`esc()` / `escapeRe()`). Every other candidate turned out to be **intentional, type-enforced, or test-pinned separation** — the input/output layering, the `LexLang` vs `InputLexLang` boundary, and the `OUTPUT_ONLY_LANGS`/`CORPUS_ONLY_LANGS` split are all deliberate and guarded by a coherence test. The hunt also surfaced one **adjacent coverage gap** (not duplication): Georgian is missing a `diagnose` meta-alias that fr/de/es have, a small forced-English leak for `ka` players. Recommendation: consolidate the one duplicate; treat the Georgian gap as a separate pro-player fix.

### Important Issues

#### [I1] Identical regex-metacharacter escape function defined twice
- **Canonical concept:** Escape regex metacharacters so a runtime string can be embedded literally in a `RegExp`.
- **Duplicate locations:**
  - `src/llm/scene/tracker.ts:13` — `esc(s)`
  - `src/translate/match.ts:30` — `escapeRe(s)`
- **Why these are semantically duplicate:** Byte-for-byte identical bodies:
  `return s.replace(/[.*+?^${}()|[\]\\]/g, '\\$&')`. Same metacharacter class, same replacement, same purpose (build a safe `RegExp` from data — noun-phrase word-boundary matching in `tracker.ts:87`; object-name alternation in corpus template compilation at `match.ts:68`).
- **Important differences:** None in behavior. Only the name and the module differ. They sit on **opposite sides** of the input/output subsystem boundary, so neither currently imports the other.
- **Impact:** Low-severity but real divergence risk: if a maintainer hardens one (e.g. adds Unicode-property escaping, or fixes a metacharacter omission) the other silently keeps the old behavior, producing a subtle `RegExp`-construction bug on whichever side was not updated. This is a textbook "two peers that will drift" duplicate.
- **Suggested consolidation:** Extract `escapeRegExp(s)` into a shared module (e.g. `src/shared/regex.ts`), import it in both call sites, delete both local copies. Pure function, no state, no coupling created — the two subsystems depend *down* onto a shared util rather than onto each other.
- **Confidence:** High (100% — verified identical against current source).
- **Found by:** Semantic Equivalence specialist; confirmed by Verifier and by direct inspection.

### Suggestions

- **[S1] `OUTPUT_ONLY_LANGS` vs `CORPUS_ONLY_LANGS` share an identical derivation expression** — `src/llm/types.ts:25` and `src/translate/corpus/index.ts:24` both compute `new Set(INPUT_LEX_LANGS.filter(l => !LEX_LANGS.includes(l)))`. **Leave as-is.** The two exports are *deliberately* separate (input-layer model-suppression vs output-layer corpus-only degradation), both files document the intent, and `src/llm/types.test.ts` pins their equality and the `CORPUS_ONLY_LANGS ⊆ OUTPUT_ONLY_LANGS` invariant. They cannot silently drift, and the separation is the seam that lets "has no LLM" split into input-only/output-only later. No action; the contract test already covers it. (Verified confidence 90%, but verdict is *keep*.)

- **[S2] `byLang()` English-fallback lookup vs an inline `?? .en`** — `src/llm/notices.ts:30` extracts `m[lang] ?? m.en`; `src/llm/prompt.ts:216` writes the same shape inline as `FEWSHOTS[language] ?? FEWSHOTS.en`. **Leave as-is.** These are the *only* two `?? *.en` sites in the subsystem (`src/llm/progress.ts:104` `ETA[lang]` has **no** fallback — it relies on complete coverage, so it is *not* the same policy). One idiomatic inline use does not justify a shared helper; the "duplicated policy" framing is overblown. (Verified confidence 60%.)

## Adjacent Coverage-Gap Findings (NOT duplication)

These are *missing parallel entries*, not duplicated logic. Recorded here because CLAUDE.md treats forced-English for a no-LLM `ka` player as a real player-harm concern, and because the hunt surfaced them.

- **[G1] Georgian is missing a `diagnose` meta-alias (CONFIRMED).** `src/llm/meta.ts:31` lists `diagnose` in `META_COMMANDS`; `fr.core.ts:323` (`diagnostic`), `de.core.ts:384` (`diagnose`), `es.core.ts:421` (`diagnostico`) each map a native word to it — but `src/llm/lexicon/ka.core.ts:205` `metaAliases` has **no** `diagnose` entry (it has inventory/save/restore/quit/score/look/odysseus/ulysses). A Georgian player on Zork I who wants the post-combat health report must type the English word `diagnose` (it still works via the `isMetaCommand` English fallback, so this is a *forced-English leak*, not a hard failure). Low frequency, but exactly the "deterministic path exists for one language and not `ka`" pattern CLAUDE.md calls out. **Severity: Important. Suggested fix: add a Georgian word (e.g. `დიაგნოზი: 'diagnose'`, plus any natural variant) to `ka.core.ts` metaAliases, with a test asserting parity.** This is a clear, low-risk, pro-player change — flagging rather than auto-applying per the report-only nature of this hunt.
- **[G2] `ka` has a smaller meta-alias set than fr/de/es — expected, not a bug.** The richer fr/de/es sets are partly papered over by their LLM fallback; `ka` deliberately supplements the English `isMetaCommand` path. Only the specific *missing* entries (G1) are actionable.
- **[G3] "CONFIRM_AFFIRMATIVE/NEGATIVE asymmetry" — could not reproduce.** The Verifier could not locate the structure the Divergence specialist described in the lexicon/types. `CONFIRM_AFFIRMATIVE`/`CONFIRM_NEGATIVE` do exist at `src/llm/inputTranslate.ts:505-517`; whether fr/es colloquial yes/no coverage is genuinely behind de is a *content* question for UAT, not a code-duplication finding. Left unverified.

## Consolidation Strategy

- **Extract canonical utility (do this):** `escapeRegExp(s)` into `src/shared/` shared by `scene/tracker.ts` and `translate/match.ts` ([I1]). Safe migration: (1) add the shared fn with a unit test covering the metacharacter set; (2) import it in both files; (3) delete both local copies; (4) run `make test`. Pure-function move, no behavior change, no new coupling.
- **Leave intentionally:** the `OUTPUT_ONLY_LANGS`/`CORPUS_ONLY_LANGS` split ([S1]), the `fold`/`directions.normalize` subset relationship ([B] — optional cross-reference comment only), the `ka`-optional language maps, and the input/output normalizer/parser separation. The contract test in `types.test.ts` is the right guard for the lang-set pair.
- **Separate pro-player fix (not part of this hunt's consolidation):** add the Georgian `diagnose` meta-alias ([G1]).
